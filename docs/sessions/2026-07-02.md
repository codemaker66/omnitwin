## Session 4 — twin walk correction: photographic panos + Street View movement

Blake verdict: "horribly broken … pure lidar … not usable" (blackened
windows, wrong interaction idiom). Root causes owned + fixed:

1. **Wrong pixels.** extract_all.py splatted lidar point RGB — never opened
   the E57 `images2D` section, which holds 149×6 photographic "Skybox"
   images (4096², exact 90° pinholes, posed). New extract_photos_v3.py
   (F:\E57 + copy in tools/twin-forge/e57-scripts) does true spherical
   resampling through each photo pose. v1 (rot90 NCC matching) and v2
   (pose snapping) were mathematically wrong — rig rotations are
   continuous (5–15° off-axis), so whole-image ops cannot be correct.
2. **Interaction.** Street View movement layer (2f05df12): click-anywhere
   floor reticle travel, WASD camera-relative walking, arrival facing the
   direction of travel; rings demoted to faint sweep discs.
3. **Data corruption in the export itself:** sweeps 1, 40, 112, 115, 145
   carry wrong per-photo roll quats (patchwork/upside-down faces).
   Automated oracles (masked NCC vs lidar pano, seam-gradient energy)
   were each fooled; the only unfoolable acceptance gate is READING TEXT
   at full resolution. Repair = per-photo solo bands → eye-judged rolls →
   `--force-rolls` → full-res text verification (protocol proven on
   scan_040; agent completing the remaining four + repo commit of v3).
4. **Ops gotchas recorded:** MemoryError-killed pye57 processes stay
   resident and exhaust Windows commit (16 GB recovered by killing
   zombies); chunked fresh-process extraction is the stable pattern.

Trigger-verifier (pano NCC, yaw-shift invariant): 144/149 sweeps good at
median 0.669. Bundle re-forged from cubemaps_photo_v3 (photographic).
Harness: all steps green, zero console errors, real glass in the windows.

### Session 4 CORRECTION (postmortem of the repair arc)

The five "corrupt" sweeps were NOT corrupt in the E57 — rig-pattern parity
(R_scan^T R_img vs healthy donors) deviates 0.0 deg on all of them. The
corruption was inflicted by v3's own auto-repair: its lidar-pano NCC
trigger has wall-band blind spots (healthy sweeps score 0.41-0.50) and its
"repairs" optimized the metric that flagged them. Fix: forced zero rolls
(raw poses); auto-repair is now opt-in (--auto-repair) and --force enables
uniform re-extraction. scan_002 = sixth polluted output, re-extracted in
the same pass.

Also discovered (agent, healthy-control test): the VIEWER renders all
panos horizontally mirrored - chirality bug in the face-upload/sampling
chain, visible only on zoomed text (scan_039 frieze, scan_145 signage).
FACE_TO_CUBE was calibrated for orientation, not handedness. Fix in
flight: shader/table chirality recalibration with a text-zoom gate added
to the harness.

Standing lesson recorded in memory: an auto-repair may never rewrite data
on the say-so of the same oracle that flagged it; full-res text is the
only vision oracle that cannot be fooled by rotations/mirrors.
