## T-441 — Fix hallkeeper offline progress sync (silent data loss + poison messages)

Request (Blake): "find something to improve to s+ tier engineering standard."

Squad role: offline / data-integrity reliability specialist.
Branch: `fix/hallkeeper-offline-sync-dataloss` (off `master` @ `2b84c7b`). Not pushed.

### What was wrong (verified against current code)

The hallkeeper tablet queues failed checkbox PATCHes to IndexedDB and replays
them on reconnect. Two real defects in that replay path:

1. **Silent data loss.** The flush built its "server" oracle from the optimistic
   local `checks` state — which already reflects every queued toggle — so
   `opsStillNeedingReplay` saw server == desired for every op, classified them all
   as already-converged, and `ackProgress`'d (deleted) them **without ever issuing
   the PATCH**. On reconnect the pending badge drained to 0 (looked synced) while
   the server never received the offline edits — the exact failure the queue exists
   to prevent.
2. **Poison messages.** Terminal 4xx replies (deleted config, revoked access, bad
   row) were treated as retriable, so such an op re-fired on every mount + `online`
   event forever and the badge never cleared.

The server `PATCH /hallkeeper/:configId/progress` is a non-idempotent toggle
(`packages/api/src/routes/hallkeeper-sheet.ts:229`), which is why blind replay is
unsafe and the authoritative-oracle approach is required.

### Fix (web-only)

- `packages/web/src/lib/progress-sync-queue.ts`: added pure, fully-tested helpers
  `partitionReplay` (replay vs already-converged against authoritative server
  state), `isReplayStatusTerminal` (4xx except 408/429 → terminal), and
  `resolveReplayDisposition` (ack on success/terminal-drop, keep on
  network/5xx/408/429).
- `packages/web/src/pages/HallkeeperPage.tsx` flush: fetches **authoritative**
  server progress as the oracle and bails (keeps the queue) if it can't read it;
  drops already-satisfied ops with no network call; re-issues exactly one toggle
  for the rest; acks/keeps per disposition; `flushInFlightRef` guards concurrent
  flushes; effect deps narrowed to `[configId]`.
- `handleToggle`: reverts the optimistic UI on a terminal rejection instead of
  queueing a doomed op.
