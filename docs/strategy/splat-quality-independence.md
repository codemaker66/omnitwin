## 1. The question and the direct answer

The question: XGRIDS gates "HD Enhancement" — fusing our own high-resolution photos with our own PortalCam scans — behind a paid tier. Can we build that capability ourselves and use our own data in our own application?

**Yes — contingent on one cheap audit (T-500), and most of the machinery already sits in this repo, unexecuted.** `venviewer_training/` contains a complete gsplat trainer (MCMC densification, antialiased mip filtering, 3DGUT distorted-camera support, bilateral-grid appearance correction, depth supervision, SH degree 3 — Config B, locked by D-006a) targeting RunPod A100 per D-016. `state/training_runs.jsonl` is empty: zero runs, ever. T-001 has been blocked on RunPod console setup (template + secrets, an owner-side action of roughly an hour) since late April — about 11 weeks, which the shepherd protocol requires this session to surface. The practical gate to independence is not XGRIDS's paywall; it is T-001.

What is missing is small in code volume: (a) an ingestion bridge from XGRIDS outputs into COLMAP form, (b) a photo-registration stage for DSLR images, (c) an executed, evidenced pipeline. Precision matters here: the part that exists (trainer + RunPod lane) is the commodity core; the unbuilt integration layer is where the feasibility risk actually lives. Sections 5 and 10 scope it, and §5's estimates apply only if T-500's frame-extraction go/no-go passes.

Three corrections to the premise, from verification (§2):

1. **The price.** No public source supports ~$800/yr. HD Enhancement ships inside LCC Studio **Premium at $2,500/yr** (XGRIDS's own US store; two resellers corroborate; an EU reseller prices ≈€1,000 base + ≈€1,000 HD add-on). If $800 came from a private quote, confirm it in T-500; either way it does not change the strategy below.
2. **The prison has an open door at the processed layer.** XGRIDS open-sourced the LCC/LCC2 container format in Nov 2025. PlayCanvas `splat-transform` (MIT) reads LCC and LCC2 natively. Third parties (MultiSet) document extracting `point_cloud.ply` + per-frame `poses.json` from LCC project output. PLY export exists below the Premium tier. Our captured splats and poses are extractable today; per-camera *frames* turned out NOT to be extractable at the raw layer — they live only inside the proprietary `.xbin` container (T-500 field result; see the §5 addendum).
3. **The genuine lock is one layer down.** No third-party tooling was found that parses the PortalCam **raw pre-reconstruction container** (raw LiDAR/frame telemetry). Base SLAM reconstruction currently requires LCC Studio — but its base tier suffices for that step. "Full independence" therefore has stages, not a switch (§8).

## 3. First principles: where detail comes from

Sharpness is information. A renderer cannot display information that was never reconstructed, and a reconstruction cannot contain information that was never captured. Exactly four levers exist, and every technology in the ChatGPT list is one of them:

1. **Capture more information** — high-res photos, slower/closer scan loops, better lighting. The only lever that creates *real* new detail.
2. **Lose less of it** — reconstruction settings, training quality, compression lineage (PLY → one runtime generation, never re-encode chains — D-013's posture).
3. **Render what you have better** — Spark settings, LOD budgets, pixel ratio, sorting. Cheapest lever; zero new information.
4. **Synthesize plausible detail** — generative models (ArtiFixer class). Never measurement-authoritative; D-012 (status: proposed) already specifies the governance completely.

The Reception Room's softness has not been localized to a lever yet. That is why diagnosis (§4) precedes all spending — the ChatGPT report is right on this point, and it is D-011 (confidence budget) thinking applied to appearance.

## 5. Building our own HD Enhancement

What XGRIDS Premium actually does: register 20–1,000 user photos into the scan's coordinate frame (control points supported) and re-optimize the splat with that added supervision. Every component of that has an open, commercially-licensed equivalent — and our ADR set already selected most of them. One status honesty note applies throughout: D-004a, D-005a, D-006a, D-012, D-013, D-015 are **proposed**; D-014, D-016, D-019, D-024 are **accepted**. Running this lane produces the ratification evidence the proposed set is waiting for — and per the CLAUDE.md rule, nothing below locks a proposed ADR into production before it is promoted.

**Architecture (three stages, only the first two are genuinely new):**

1. **Bridge (new — T-501).** LCC project output → COLMAP dataset. Inputs: `poses.json` (per-frame timestamp/translation/quaternion), extracted camera frames (from the raw project per T-500's audit; fisheye 200° and front 100° cameras), and the LCC splat/point cloud for metric anchoring. Output: `images/` + `sparse/0/{cameras,images,points3D}.bin` in the exact `colmap_v2` layout `simple_trainer_depth.py` already expects. Tiled LCC2 is handled by `splat-transform` merge (MIT, reads LCC2 natively). Scale/frame lock in two recorded steps, named precisely: pose-based similarity alignment (`colmap model_aligner` with reference positions) and cloud-to-cloud ICP against the LiDAR-derived cloud (Open3D/CloudCompare — the same move `project_e57_depth.py` already makes for E57), each persisted as a `TransformArtifactV0` (D-024) carrying a full residual report — mean, median, RMSE, p95, overlap percentage, control-point error — plus fixed-view visual validation. No universal residual gate is pre-committed before pilot evidence: purpose-fit thresholds are proposed from T-500/T-505 results (D-003's <5mm RMS stays as written for the projective-texturing use-case it belongs to; D-011's confidence tiers grade everything else). Frame extraction is the go/no-go: the poses+splat extraction path is verified third-party practice; readable per-camera frames are not yet verified (T-500), and the effort estimates below apply only if frames pass. Relation to prior art: T-385 already green-lit the splat-transform bridge for *post*-training packaging; T-501 is its *ingestion-side* sibling.
2. **Photo registration (new — inside T-503).** Shoot DSLR/phone photos per a capture protocol (fixed WB/exposure, low ISO, heavy overlap, multiple heights; 200–400 general + 20–60 per hero feature — fold into the T-203 Capture Control Network vocabulary and D-015 tier discipline). Localize them against the bridge model with COLMAP's documented continue-from-existing-model path (`image_registrator`, or `mapper --input_path` for refinement) or hloc (Apache-2.0). Mixed camera models are native to COLMAP — each image carries its own model (OPENCV_FISHEYE for scanner frames, PINHOLE/OPENCV for DSLR); rig machinery is unnecessary since the DSLR is not bolted to the scanner.
3. **Training (already built — Config B).** The existing trainer is *unusually well-suited* to exactly this job, which is the strongest "unknown known" this analysis surfaced: **3DGUT** (already flagged on in Config B per D-006a "distorted-camera ingestion") is the strongest available fit for the fisheye frames — with the honest caveat that 200° FOV exceeds the ~180° validity envelope of COLMAP's fisheye parametrizations and 3DGUT's commonly tested range, so the pilot must confirm registration/reprojection before this is assumed; fallbacks are cropping fisheye frames to a sub-180° region, or starting from the two 100° front cameras; the **bilateral grid** (already on) is per-image appearance compensation — the designed mechanism for absorbing exposure/WB mismatch between scanner frames and DSLR photos taken on a different day (unproven on this data until T-503; and because per-image appearance optimization can flatter held-out PSNR, T-502/T-503 evaluate with grids frozen on holdout views and lean on LPIPS plus the photographic bar); **MCMC** (already on) re-densifies where the new photos add information. Depth supervision is optional for v1: PortalCam's LiDAR-derived depth availability is a T-500 known-unknown; `project_e57_depth.py` generalizes if a metric cloud exists, and the trainer runs without it.

**Two initialization modes, both planned:** (a) full retrain from bridge data (clean, default; the D-006 eval protocol applies unchanged); (b) warm-start from the LCC PLY and refine with new photos only — cheaper, but the continual-refinement literature (CL-Splats ICCV'25, Cross-Temporal 3DGS AAAI'26, GaussianUpdate) documents the failure modes: catastrophic forgetting of uncovered regions, frozen/under-densified primitives, session exposure mismatch. Mitigations: freeze or heavily regularize Gaussians outside the photo-covered region (this is D-005a's residual-extraction machinery pointed at a new purpose), rely on the bilateral grid for appearance, and always compare against mode (a). Hero micro-splats — ChatGPT's most creative suggestion — are not a new idea here: they are D-005a (view-dependent residual layer) + D-024 (Hero Region Specialists, with fireplace/chandelier/carved-wood already enumerated) + T-024 (automated residual extraction), composed. Spark renders base + residual layers together natively.

**Output and packaging:** D-014 signed bundle (`scene.ply`, `training_config.json`, metrics, manifest) unchanged; runtime SOG/SPZ via splat-transform with single-generation lineage (master PLY → runtime, never runtime → runtime; SOG's own spec warns re-encoding corrupts).

**Evaluation:** the D-006 protocol (held-out view set, PSNR/SSIM/LPIPS, ship-gate deltas) plus the Genjutsu photographic bar at T-500's fixed viewpoints. First benchmark is our retrain vs. the LCC2 export we already have (T-502); the photo-fused pass then has an honest baseline (T-503).

**Effort and cost, honestly estimated:** T-501 bridge 3–5 focused sessions (poses.json → COLMAP writer, frame extraction per audit findings, alignment + TransformArtifact, fixture tests); T-502 baseline run + eval 2–3 sessions; T-503 shoot + registration + fused run + eval 3–5 sessions. Compute per D-016: $5–15/routine run, $30 ceiling; a full Reception Room program (~6–10 runs with ablations) ≈ $30–150. Priced honestly, 8–13 focused sessions dwarfs $2,500/yr in pure dollars — the case for building is not subscription savings. It is custody, unlimited and per-hero enhancement passes, D-012 provenance semantics no vendor export carries, the S+ ownership doctrine, and the fact that the trainer/RunPod/eval spend is already sunk and mandated by D-014/D-016 for reasons independent of XGRIDS. Precondition for all of it: **T-001** (Blake: RunPod console template + secrets, then the mandatory $0.20 smoke test).

**Honest uncertainty (Law 1):** LCC's trainer is tuned to XGRIDS's own calibrated hardware; parity on the first attempt is not guaranteed, and this doc does not claim we will beat their base reconstruction. The claim is narrower and defensible: photo *fusion* on top of a good base is standard photogrammetry + the trainer we already own, the licensing is clean (§7), and the bake-off (T-502) decides with evidence, not opinion. If our retrain lands clearly below the LCC2 export, Stage 2 still works — fuse photos as a *residual/hero* enhancement over XGRIDS's base rather than replacing the base (the layered architecture makes both compositions legal).

**T-500 field results (2026-07-16 addendum):** the raw-project audit answered the go/no-go — frames exist only inside the proprietary `.xbin` (XBAG) container, and the factory calibration YAMLs inside `lixel.zip` are password-encrypted. Consequence: initialization mode (b) — warm-start from the LCC master PLY with registered DSLR supervision — is the **primary** HD lane; mode (a) scanner-frame retrain moves to Stage 3 (raw-container parsing + the §7 legal read). Trajectory is open (10 Hz `poses.csv` in the project, 5 Hz `poses.json` in the LCC2 export), so the pose half of T-501 is confirmed feasible. The LCC2 runtime was dissected: SH3 is present but palettized to a 65,536-pattern k-means codebook per tile, `antialias:false` (Spark blur compensation must match), built model 336.7 MB vs 63.5 MB shipped (5.3× post-build compression), `hdImageCount:0`. One cheap probe stays open before calling mode (a) closed: LCC Studio's build workspace on C: may cache extracted frames. Full evidence: `docs/operations/portalcam-raw-audit-2026-07-16.md`.
