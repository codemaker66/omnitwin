## §0 Thesis

Venue teams call their booking system "the diary." The product is **a living Diary**: one truthful surface where a date moves pencil → ink → show-day → settled, and sales, planning, ops, client, and suppliers see facets of the same object. Every industry failure (stale run sheets, double entry, ghosted holds, version drift) is fragmentation of the diary across email/Excel/paper/heads. The product wins by making fragmentation structurally impossible. Pitch: **"Pencil it in. Ink it. Run the day. Nothing goes stale."** Lifecycle: **Sell → Hold → Plan → Run → Settle → Learn.** Design language: quiet luxury command centre — already the house style (graphite/gold/cream/cyan tokens in `web/src/global.css`).

## §1 Status model & vocabulary (R1, R6, repo)

**Three axes, three carriers** (validated by Momentus's booking-vs-contract split; maps onto existing repo tracks):

| Axis | Carrier | States |
|---|---|---|
| Sales | `opportunities.stage` (exists) | new → … → won/lost |
| Deal/contract | `proposals.status` (exists) + quotes | draft → sent → approved → signed |
| Commitment | **`bookings.state` (NEW)** | prospect → hold(ranked) → ink; exits: released, expired, lost, cancelled |
| Ops readiness | `events.status` (extend) | draft → in_planning → ready_for_ops → completed |

"Live today" is **derived** from clock+phases, never stored.

**Canonical vocabulary (locked, from R1/R6):** pencil (hold), ink (definite), 1st/2nd option, **joint first option**, decision date, challenge, release, avails, prospect (non-blocking), provisional (community-let display term). Display labels are **per-venue configurable** (Tripleseat's rigidity is a named complaint; Momentus renames to "pencil" — the metaphor is industry-native). Licensing vocabulary is **keyed to `venues.jurisdiction`**: Scotland = occasional licence / extended hours; England & Wales = TEN. Never conflate (R7).

## §2 Domain model locks

1. **One `bookings` table**, kinds `prospect | hold | ink | internal_block`. Prospects never block and may overlap (they feed demand heat). Blocks generated by rules (blackouts/maintenance). Fields incl. `spaceId`, `eventId` (nullable), tstzrange, `rank` + `jointFlag`, `decisionAt`, `ownerUserId`, `seriesId` (nullable, day one).
2. **Hard floor:** Postgres **exclusion constraint** (btree_gist, raw SQL migration — Drizzle can't express EXCLUDE) — two `ink` bookings can never overlap in one space. Holds overlap by design. Everything else advisory.
3. **`event_phases` gains `spaceId`** — the keystone migration. Phases become the **Occupancy Footprint** (load-in → setup → live → flip → teardown → load-out) rendered as segmented sub-blocks. Buffers/turnaround rules are templates that generate/validate phases, not a parallel system. Mid-event flip = internal phase; between-event flip = turnaround gap check.
4. **`events` gains `clientAccountId` + `opportunityId`** (today clientName is a bare string) and the **headcount triple: guaranteed / expected / setFor** (R2).
5. **Rooms:** hierarchy + combinability on `spaces` (booking combined blocks children); capacity stays at configuration/layout level — Fit Finder joins configurations.
6. **Resources:** shared singletons (goods lift, PA, kitchen, portable bar) bookable against phases. `asset_definitions` covers furniture; singletons are new.
7. **Payment milestones = flexible rows** (arbitrary schedules incl. monthly plans), preset per venue type (exclusive-use / dry-hire / marquee / restaurant) (R3).
8. **Caterers = suppliers with a trade** + venue↔supplier relationship (approved/preferred, rank, contracted flag). Trades Hall preset: Regis Catering, Top Class Catering (preferred). Event carries `serviceStyle` (buffet | plated multi-course | …). Client portal selection order: venue-preferred → venue-approved → bring-your-own (auto-creates supplier + load-in needs). Locale ratings directory = P4 network feature seeded from debrief scores.
9. **Reuse, don't duplicate:** tasks = union of `follow_up_tasks` + `ops_tasks` (no third system); audit = `general_audit_log` + per-entity status histories (add `event_status_history`, booking history, following existing convention); diffs/notifications = `event_plan_changes` + `event_plan_notifications` (exists); supplier links/acks (exist); `external_calendar_links` (exists). New state machine `api/src/state-machines/booking.ts` follows house pattern.
10. **Timezone:** UTC + `venues.timezone` (exists, Europe/London); all rules evaluate venue-local; mandatory tests straddling March/October transitions (R8; s.66 counts hours after midnight).

## §3 Option Ladder & hold policy (R1, R6 + owner decisions)

- Ladder depth: **unbounded in schema, per-venue `max_hold_rank`** (owner chose configurable; propose default 3; The GRAND runs 4).
- **Joint first from day one** (owner). Semantics: shared rank 1 with explicit joint flag; offer states "first to confirm wins"; ink atomically wins the race (exclusion constraint arbitrates); loser's coordinator gets kind notification + follow-up task; a challenger ultimatums **all** joint firsts simultaneously.
- **Hold hygiene (the wedge — nobody mid-market ships the trio):** every hold requires decision date + owner + next action at creation; reminders **T-7/T-3/T-1**; auto-lapse unless deposit/contract attached; **auto re-sequencing on release with a human ping** ("MacLeod wedding is now 1st option — tell them"); "Open Tentatives" aging panel + Monday briefing.
- **Challenge:** default **48h** (24–72 range, business-days option); triggered manually or by displacement status-conflict rule (§5). Ships **P1** (owner: contention = "sometimes, peak Saturdays" → high value density on top revenue dates).
- **Lead-time hold-duration matrix (defaults, editable):**

| Lead time | Corporate | Wedding/social | Hot date |
|---|---|---|---|
| >12mo | 14–30d | 14–21d | 7–14d |
| 6–12mo | 14d | 14d | 7d |
| 3–6mo | 7d | 10d | 72h |
| 1–3mo | 72h | 7d | 48h |
| <1mo | 24–48h | 48h | 24h |

- **Dynamic policy:** hot band (>80%) auto-shortens one tier + 24h challenge; chronic cold (<20%) relaxes. UK anchors: 10 working days (Islington), 14 days (The GRAND).
- **Wash weights** for demand forecasting: prospect 20–30%, hold 60–80% (calibrate to history).

## §4 Conflict engine (R1, R4, R6, R7)

- **Pure deterministic function** over (bookings, phases, rules, resources, assignments, review states) for a visible range → typed conflicts with **severity tiers (blocking / warning / info)** + plain-English explanation + inline resolutions. Computed on demand P0; persisted on mutation later.
- **Soft vs hard** (R4/Resource Guru): hard = ink double-book (DB-prevented); soft = everything else. **Clash dialog** names the conflicting booking in plain language, offers: waiting list (→ waitlist is P1), override-with-reason, cancel. **Elastic auto-resolution**: clearing the underlying clash auto-releases what it blocked.
- Conflict types: room double-book; insufficient setup/teardown; flip conflict; layout not reviewed (Truth Mode gate chips share the visual language); proposal not approved; guest count vs layout capacity; supplier load-in overlap; resource singleton clash; staff conflict; blocked/maintenance; missing handoff pack; stale evidence; **noise-sensitive adjacency** (soft matrix per room pair); curfew overrun vs operator-entered terminal hour (planning flag ONLY).
- **Honesty pattern:** mirror repo's `not_checked` defaults; copy always "planning support / human review"; **never a compliant/OK badge near licensing/security fields** (R7: "trigger to change: none").

## §5 Yield & revenue (R6)

- **YHI (0–100)** per room×date×day-part = 0.5·ValueScore + 0.5·DemandScore (weights exposed). Value = expected contribution ÷ (saleable m² × day-part hours **incl. setup/teardown**), normalised per-room vs own history. Margins (configurable): room hire .90, F&B .33, AV .70. Contribution default; revenue fallback if margins unset. Hover explains (ProPAST, band, pace).
- **Bands (Cornell):** <20% cold · 20–64 neutral · 65–80 warm · >80 hot.
- **Displacement warning:** slot ≥65% band AND candidate contribution < forecast displaced contribution → RED (hot+negative) / AMBER (warm, or whole-hall buyout beats divided aggregate) / GREEN; always show count+value displaced; widen/suppress at long lead. **Status-conflict rule:** low-value hold blocking higher-value enquiry → prompt a challenge (wires into §3).
- Day-parts default: Morning/Noon/Afternoon/Evening/Overnight.
- **CMA resolution (R3×R6 tension):** yield expressed via **price tiers, never deposit size**; deposits stay modest per CMA in every template.
- Positioning stats bank: Saturday 47% of UK weddings (2025, first sub-half); 79% book within 4 weeks of first contact; 72% won't enquire without visible pricing; only ~16% of UK venues offer live availability; Scotland avg wedding ~£17.8k.

## §6 Client journey, portal, payments (R3)

- **#1 complaint = radio silence in the long middle; root cause often coordinator turnover.** Product answer: **team continuity surface** — every event shows "your team"; owner reassignment auto-drafts intro email + portal confirms all details on file. Drip rule: **never >6 weeks silent** mid-journey.
- **Drip (14 touches, defaults):** 1h auto-ack w/ transparent VAT-inclusive pricing → 24h named human → tour+24h recap+quote → booking welcome pack + portal invite → +1wk meet-the-team → 6–8wk check-ins → 9mo & 6mo payment touches → staff-change trigger → 10–12wk menu pack → ~6wk book final-details meeting + checklist → 4–5wk balance reminder → 2wk final numbers + reassurance → day-before warm msg + day contact mobile → post-event thanks + **private-first feedback** (before public review ask).
- **Payment defaults (CMA-aware, presets per venue type):** modest booking deposit + graduated cancellation scale (Bijou ceiling ~28–37% near date, net of re-let); 50% of remaining at 6mo; balance 4wk; **final numbers 14d** (2–4wk configurable; increases ok, reductions non-refundable); menu 12wk; details meeting 4–6wk; table plan final 2–4wk; dry-hire: CMA expects generally **no** deductions. Standing "seek legal advice" disclaimer on all term templates. Instalment plans supported.
- **Portal:** **magic-link auth** (reuse share-token pattern; "another password" = top client friction); never empty on first login; condensed client timeline; messaging anchored to items (extend `proposal_comments` pattern); payment schedule w/ auto reminders. **Skip:** guest website/RSVP builder. Dietary is a standard pipeline step (UK 2026: 4.2 per 100 guests) → kitchen board.
- **Deliverability = P1 infrastructure** (challenge/reminder emails in spam is a named complaint): venue-domain sending, SPF/DKIM/DMARC in onboarding (Resend supports).

## §7 Run surface (R2, R7)

**Day Sheet (one page, self-generating from the Diary; diff-aware; regenerates on change):** header (event, date, **version stamp + unique ref + "on today" name**); times column (load-in → arrival → moments → service → **curfew cluster** → lock-up); headcount triple; room states + flip triggers; contacts (client on-site, suppliers + arrival windows, duty mobile, **keyholder/ARC**); access & building (keys/codes, **alarm zones to unset/reset — prefilled from footprint, operator-confirmed**, heating-on time, fire-exit check, dietary flags, listed-building/décor restrictions, **door/bay owner**); diagrams/menus as linked attachments. Convention: **"Final" by T-72h** + briefing nudge. Design test: staff cross-referencing >1 document = failure. Variants: ops (full), kitchen board (numbers/dietaries/service times, wall-mountable), client summary, **hirer pack** (self-service community lets: table storage, lock-up sequence, keypad scramble). Print: gorgeous A4 + QR to live version.

**Curfew cluster (all operator-entered):** terminal hour, last-entry, music-off/wind-down (30–60min before), drinking-up (15/30min), outdoor terminal. Non-blocking helper text; standing disclaimer; occasional-licence **lead-time task** ("Glasgow board ~7 weeks") as planning support. Fire-detector isolation = paired checklist (isolate w/ building-mgmt coordination → **reinstate at lock-up**, fire-watch interim).

**Flip Planner defaults:** 60min window (30–45 practised target), hard green-light owner+time, pre-stage, **break back-to-front / build edges-inward**, service-entrance routing, +1–2 float crew beyond service team, guest-holding plan, cake-cut lever; escalation >150–200 guests or multi-event day. Actuals recorded → learned durations (P4 estimator).

**Mobile ops mode:** offline-first (service-worker cached Day Sheet + sync queue), one-handed, glove-friendly, large targets, persistent "what room / what state / what time / who to call," room state chips (clean/set/live/dirty/flipping) one-tap, quick broadcast degrading to text. **We do not build radio/chat.** Failure catalogue (double-book, stale sheet, unbriefed staff, slippage, no-show supplier, keys, heating) = acceptance test scenarios.

## §8 Interaction spec (R4, R5)

- **Board:** horizontal time axis, rooms as lanes; sticky room column; collapsible groups (floor/building); per-lane capacity+utilisation headers; **holding tray** (unassigned enquiries/holds → drag to lanes); day/week/month zoom — day zoom = hour-granularity axis where several bookings per room per day is the normal case, sequenced by footprints with turnaround gaps visible between them; zoom-dependent disclosure priority **colour > title > time > icons**; column-packing for concurrent bars; all-day band; hatched closed days; now-line. Overlay model only for find-a-free-room compare view.
- **Drag:** 15-min snap to **absolute quarters**; Shift = 1-min free; ⌘⌥/Ctrl-Alt-drag duplicate; **5px** activation; edge auto-scroll; ghost shows lane + start–end + live conflict colour; Esc-cancel + cursor time label are **deliberate custom builds**; invalid drops animate back. **State-gated drag:** pencils drag freely, **ink resists** (explicit intent required). Mobile: long-press **250ms** lift + haptic; handles appear after lift.
- **Keyboard:** APG Grid model; `n g t d w m 1–9 ?`; arrow reassignment across lanes/time; Enter/Esc universal; **⌘K palette** = commands + NL parse ("Wedding reception Grand Hall Saturday 6pm–midnight 120 guests hold" → room/type/time/duration/headcount/status). **Keyboard drag ships day one** (dnd-kit KeyboardSensor + live-region announcements) — exceeds FullCalendar (open since 2015) and Bryntum.
- **Feel:** interactions **<100ms**, no spinners after first load; **undo-toast ≥5s** over confirm dialogs; animations 0.1–0.25s asymmetric, origin-referenced; optimistic with honest rollback ("that slot was just taken"); haptics on lift/drop/confirm. Pencil texture on holds; **ink-fill animation** on confirm.
- **Mobile split:** clients = agenda/DayTicker + bottom sheets; floor staff = offline day list + big status controls; no dense grid on phones.

## §9 Technical architecture (R5, R8, repo)

- **Timeline: BUILD, DOM-first, no canvas.** TanStack Virtual two-axis (lanes × time); absolute+transform blocks; live DOM in low hundreds; `contain: layout paint` on lanes; `content-visibility` on collapsed groups; `will-change` only while dragging; `useTransition` for zoom/filter never drag. Budget: 60fps drag; day view <300ms; CI worst case 40 lanes × 2,000 blocks via existing frame-budget harness.
- **Realtime: server-authoritative commands over existing `@fastify/websocket` on Railway** (zero new vendors; `ws/auto-save.ts` precedent). Mutations = commands validated in a Neon transaction (exclusion constraint = final arbiter) → commit → broadcast. Optimistic UI + rollback. Presence + advisory soft-locks same channel (never the correctness mechanism). **CRDTs never for bookings** (Loro documents the venue double-book case verbatim); Yjs at most for free-text notes, later.
- **Print:** virtualisation breaks print → dedicated render-all print route via headless Chromium (Playwright already in repo); `@page` landscape A4; `break-inside: avoid`. pdfkit stays for data-driven docs.
- **Sync (P3):** Google + Graph read integrations; **webhooks-as-hints + 15-min delta/sync-token reconciliation + nightly full sweep**; proactive renewal jobs (Google channels ~7d, Graph 4,230min, delegated-only for /me/events); handle 410/`syncStateNotFound` anytime; dedupe (at-least-once); expand recurrences to instances, reconcile cancellations explicitly ("cancel, don't delete"); `singleEvents` flag consistency (400 trap). **ICS Stage 0 (days):** our own polite poller 15–60min (`Last-Modified`), advisory busy blocks, **never gates a paid booking**. Buffers/min-advance-notice absorb sync latency. Two-way = buy-gate: >~1 eng-quarter/provider → Cronofy (EU-hosted).
- Storage: UTC + IANA + venue-local evaluation; money in minor units (house convention); `venue_id` on everything.

## §17 Sales OS — Salesforce-class coverage, venue-native (v1.3)

**Doctrine:** cover 100% of the jobs a venue hires a CRM for; exceed each; refuse the bloat deliberately (per the original project brief: best ideas, never literal overfit). Marketing truth: "everything Salesforce does for a venue, in under 100ms, with no consultant."

**Already covered/exceeded by Canon+repo:** lead capture & routing (§16 intake + SLA timers); pipeline (opportunities + Option Ladder — dates ARE the pipeline); accounts/contacts (CRM spine); tasks (follow_up_tasks + ops_tasks); quotes/CPQ (quotes + packages + pricing_rules, capacity/layout-aware); documents (proposals + versions + share tokens); automation (drip, hold hygiene, challenge engine); reporting foundations (revenue analytics, yield, Monday briefing, ops footer); forecasting (wash-weighted pipeline + YHI); permissions (workspace roles + booking-capability ceilings); audit (general_audit_log + histories + time-travel); dedupe & merge queue (§16); consent/campaign separation (§16); search (⌘K); mobile (ops mode — field tool, not mini-CRM).

**Gaps now formally added:**
1. **Comms auto-logging (P2):** every client email in/out threads onto the client + event automatically — the R11 per-venue ingest address doubles as a BCC/log address; quick one-tap call logging for phone enquiries. Kills the #1 CRM failure (unlogged activity).
2. **Unified activity timeline component (P1–P2):** one chronological narrative per client and per event — emails, calls, tours, payments, status changes, notes — rendered from existing histories + audit log. The "client 360," fed by machinery we already have.
3. **Tours as first-class Diary objects (P1):** viewings occupy rooms, carry route notes, auto-generate follow-up tasks (was in the feature universe; now explicitly phased).
4. **Saved views + universal CSV export (P2):** every list/board filterable, saveable, exportable. Defer any drag-drop report builder; saved views + export cover 95% of venue reporting demand.
5. **Automation recipes library (P3):** curated when-X-then-Y recipes (e.g. "enquiry idle 3 days → task owner", "deposit paid → generate ops tasks") — prebuilt, toggleable, no visual flow-builder in v1.

**Universal law (elevated from repo convention):** nothing exists without a next action, an owner, and a date — enquiries, holds, opportunities, tasks. The anti-stale principle applied to the CRM layer; the Diary's equivalent of Salesforce hygiene dashboards, enforced at creation instead of reported after death.

**Anti-Salesforce principles (the "wish it had" list, as commitments):** <100ms interactions, no page loads; undo over confirmation; zero-admin day-one usability (config, never consultants); one surface (the Diary) instead of forty tabs; per-venue vocabulary instead of a customisation platform; search that finds things; role-appropriate access for kitchen/hallkeeper surfaces without CRM-seat friction (pricing model = owner decision, flagged not locked).

## §18 Ink & Gilt — visual language locks (image bake-off; revised v1.5, owner-approved)

**Name:** Ink & Gilt. One metaphor (pencil→ink), three surfaces:
1. **Command Centre (staff):** graphite/gold/cream over Swiss-grid discipline. Holds = paper-textured cards, handwritten-italic event name, one folded corner, brass pin. Ink = solid dark blocks with gold left edge. Occupancy Footprint rendered as in-block SETUP/LIVE/TEARDOWN segments (hatched tails). 2. **Print (Day Sheets, avails, hirer packs):** the Ledger — cream paper, letterpress serif, pencil-hatch vs navy ink, wax-seal conflict marks, architect title block (version, date, on-duty, drawn-by), QR captioned "Scan for live updates".
