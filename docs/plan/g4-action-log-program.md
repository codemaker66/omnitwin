# G4 — the Action log programme (T-522)

2026-07-16 · spec: 03 §1 principle 1 · gap: 06 §4 G4 ("weeks, load-bearing") · one slice = one session, card discipline applies.

## The one-sentence spec (03 §1, verbatim anchor)

> Every mutation in the product … is one typed, serializable, invertible `Action { id, actor, intent, payload, inverse, provenance, ts }`. … This single decision buys undo/redo, version history, multiplayer sync, the audit trail, AI tool-use (the Action schema *is* the copilot's tool API), phase diffs, and session replay.

## Audit — what already exists (2026-07-16)

**The hard core is built.** `packages/web/src/lib/editor-history.ts` (T-447) is a pure, dependency-free, command-sourced engine recording **invertible field-level deltas** (`HistoryDelta { added, removed, updated: ObjectFieldPatch[] }` with before/after patches), drag-coalescing epochs, selection restore, and bounded memory (100 entries / 2 MB). One timeline already spans the 3D scene and the 2D blueprint (a 42-case reducer feeds the same history). **G4 is an extension of this engine, not a rewrite** — materially smaller risk than the gap audit priced in.

**The gap between `editor-history` and the Action schema:**
1. No envelope — deltas carry no `id / actor / intent / provenance / ts`, and no serialization contract (`@omnitwin/types` has no Action schema).
2. Coverage — only placed-object mutations flow through the engine. Persisted mutations that bypass it: markup strokes (`markup-store`, own persistence schema), lighting rig (`lighting-rig-store`), event details (API-direct), guideline/measurement/bookmark state where persisted. ~~object notes (API-direct)~~ — **slice-2 correction:** object notes ride `setObjectNotes → recordedHistory` through the engine, so slice 1 already covered them (pinned in `editor-store-action-log.test.ts`).
3. The undo timeline **evicts** (100-entry cap) — an audit log must be append-only and eviction-exempt.
4. No actor model (operator vs AI vs collaborator) — the AI-tool and audit stories need provenance from day one.
5. No server emission (03 §2: the API "emit[s]/consum[es] Actions") — persistence and the audit read model don't exist.

**Deliberately out of scope for the whole programme:** UI-only stores (device, perf, toast, visibility, xray, section, camera…) are never Actions; Yjs/Zero sync stays deferred (T-105 posture) — the envelope is *designed* so a CRDT layer can adopt it later, but no sync code lands.

## Slice 3 — SHIPPED 2026-07-18

Persistence + the audit read model, both slice-3 preconditions cleared first:

- **Precondition 1 (pollution):** `JsonValueSchema`'s record branch rejects `__proto__`/`constructor`/`prototype` keys at every depth via a PRE-parse `z.preprocess` guard — a post-parse refine can never work here because zod's record parser rebuilds objects with plain assignment, where a `__proto__` key silently becomes the output's prototype. Reviewer independently traced zod 3.24.2's dirty-status propagation and fuzzed the guard: airtight.
- **Precondition 2 (parse cost):** `ActionLogBatchSchema` bounds ingestion BEFORE the recursive envelope parse — batch cap 500, per-action 64 KiB, depth cap 32 measured by an iterative `jsonDepth` (explicit stack; depth runs before `JSON.stringify` so neither recursive path ever sees a pathological chain). Practical batch ceiling is the server's 2 MB bodyLimit (documented at the caps).

Delivered: `action_log` table (migration 0059; client-supplied action uuid as pk → `ON CONFLICT DO NOTHING` idempotent retries; server `bigserial ordinal` is the only read order — client clocks never order the trail; `recorded_ts` operator clock vs `received_at` server clock kept distinct end-to-end; `submitted_by` = the AUTHENTICATED principal at ingestion, the anchor for the self-reported actor blob; config FK cascade follows the repo convention — deliberate: configuration deletion is soft in practice, and a hard delete taking its trail is the sibling-table semantic). Routes at `/configurations/:configId/actions` (POST bounded batch ingest, GET ordinal-paged audit read; `authenticate` + placed-objects-pattern config access; append-only by code contract — no update/delete surface). Web: `sentCount` flush cursor with fold-aware renumbering; `flushActionLog` ships the unsent tail on save success (anchored to the saved revision, auth path only) and at the config boundary; whole-body-fenced fire-and-forget. `verifyReplayable` gates slice-4 replay (strict ordinal order, unique ids, valid envelopes, real inverses on mutations).

Review cycle (typescript-reviewer, Block → fixed same-session, reproductions pinned): **CRITICAL** — a fold landing during an in-flight flush left the ack in a stale index frame, stranding the cursor past the shrunken array and silently orphaning the tail at routine cap-state usage (reviewer reproduced it against the real modules); fixed structurally — acks resolve the chunk's last action ID against the CURRENT array (`markSentThrough`), so the store's append-time fold adjustment stays authoritative and a folded-away id no-ops; both fold scenarios pinned (cursor-beyond-span: summary correctly stays local, constituents already persisted; cursor-inside-span: summary must ship — that's the data). **HIGHs** — read-side parse was unbounded (RangeError at ~1000 nesting levels, escaping the api client's error contract): `BoundedJsonValueSchema` now caps audit-page payload/inverse like ingestion; intra-batch duplicate action ids now rejected at the schema (accounting no longer depends on Postgres intra-statement conflict semantics); `submitted_by` added end-to-end (schema/SQL/route/read/client) closing the actor half of claim safety.
