# D-016 — RunPod-canonical training environment

Status: accepted
Date: 2026-04-26
Authors: Blake
Supersedes: none
Depends on: D-006a, D-014

## Context

D-006a (gsplat MCMC + bilateral grid with 3DGUT hedge) committed to gsplat 1.5.3 as the trainer. D-014 (Venue Artifact Factory) defined the training-runtime boundary as a signed `AssetVersion` bundle. Neither specified WHERE training executes.

Local Windows training was deprecated as of 2026-04-25 due to environment fragility: gsplat 1.5.3 only on Python 3.10, NumPy 2.x patches, fused-ssim CUDA mismatch with PyTorch 2.4 vs CUDA 13.2, OOM at ~2.5M Gaussians on RTX 4090 24GB. After deep research synthesis (two outputs, ChatGPT and Claude), Linux + RunPod A100 80GB is the only path that satisfies VRAM headroom, dependency stability, and cost ceiling simultaneously.

## Decision

RunPod is the canonical training environment for all Venviewer splat training. A100 80GB on Community Cloud on-demand is the default; Secure Cloud is used for canonical bake-off runs. The trainer image is published as `omnitwin/trainer:1.5.3-cu124` (or equivalent — internal name, not customer-facing per the cosmetic-OMNITWIN rule). Local CUDA-based splat training is deprecated and unsupported.

## Consequences

Every training run executes on RunPod. Setup overhead per run is ~2–5 minutes pod boot + container start. Per-run cost lands in the $5–15 band for routine training (3–10 hour runs at ~$1.19/hr) and a $30 ceiling for canonical multi-flag runs.

R2 is the canonical storage layer for both inputs (`colmap_v2` datasets, depth priors) and outputs (signed AssetVersion bundles). R2 zero-egress means pulling input fresh every run is cheaper than maintaining persistent network volumes.

Trust boundary: training pods receive R2-scoped credentials only, never database or Fastify credentials. Backend ingestion (T-053, deferred to post-T-018) is the only path that signs and registers bundles into the AssetVersion table.

Local development continues for everything except splat training itself — web app, API, schema work, eval analysis, dataset preparation, all stay local. Only the training run pipeline is RunPod-bound.

Operational specifics — pod template definition, secrets, runbook, smoke-test procedure — live at `infra/runpod/RUNBOOK.md` and `infra/runpod/runpod-template.json`.

## Alternatives considered

- **Local RTX 4090 training (the deprecated state).** 24GB VRAM is insufficient for a 5M Gaussian cap with the full extension stack (Mip-Splatting + 3DGUT + bilateral grid + DN-Splatter depth). Windows Python environment fragility is well-documented in session history (gsplat 1.5.3 on Python 3.10 only, NumPy 2.x patches, fused-ssim CUDA mismatch).
- **AWS g6e.xlarge (L40S 48GB).** Priced at ~$1.86/hr, more expensive than RunPod A100 Community at ~$1.19/hr, with no advantage for this workload. Considered as ablation alternative only.
- **Fly.io GPU.** Not available at A100-class. Smaller VRAM tiers only.
- **Vast.ai marketplace.** Pricing potentially lower per hour but operational overhead — verifying providers, image cache cold-starts, inconsistent driver versions — is not worth saving ~$0.20–0.40/hr at current scale.
- **Self-hosted A100 server.** Capex ~$15–20k for one A100 80GB, plus colocation/power/cooling. Breakeven >5000 hours of training, many years away at current pace.
- **RunPod Spot/interruptible.** Cheaper but only viable after checkpoint resume is proven end-to-end. Deferred until the on-demand pipeline has a multi-run track record.
