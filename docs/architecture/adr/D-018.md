# D-018 — The Product Vision: Verified Generative Venue Planning

Status: accepted
Date: 2026-04-27
Authors: Blake
Supersedes: none
Depends on: D-009, D-011, D-014, D-015

## Context

Venviewer is currently understood internally as "a 3D venue planning platform with photoreal digital twins." That understanding is correct but insufficient. It describes the substrate, not the product.

The full product is one loop, executed at S+ tier:

> A customer describes an event → Venviewer creates a beautiful, physically valid, costed, shareable plan inside a real photoreal venue → the venue can execute it.

This loop is the product. Every feature exists to serve it. The loop is the test of whether any proposed feature belongs in Venviewer.

Market signal supporting this framing: Cvent's 2024 acquisition of Prismm validates 3D event diagramming as strategically valuable. Matterport's hospitality partnerships validate immersive venue media for sales conversion. Apple RoomPlan validates phone-LiDAR capture as mainstream infrastructure. 3D Gaussian Splatting (Kerbl et al., SIGGRAPH 2023) creates the technical opening for browser-native photorealism. None of the existing players close the loop end-to-end. Venviewer's opportunity is to be the first.

## Decision

Venviewer commits to a product vision of eleven first-class capabilities, organised around the verified generative venue planning loop. The eleven are listed below with strategic priority and architectural notes.

The eleven are committed as roadmap intentions, not as build-imminent work. Each is tracked as a backlog task with explicit unblock criteria. Activation order is determined by foundation readiness, not by feature appeal.

### The eleven capabilities

**1. Verified Photoreal Twin** — already in architecture per D-015 (Capture Certification Tiers) and D-011 (Spatial Confidence Budget). Every venue ships with a tier (Bronze / Silver / Gold / Black Label), measurement metrics (PSNR/SSIM/LPIPS), confidence heatmap, scan provenance, and hardware lineage. This is the substrate the other ten capabilities build on.

**2. Truth Heatmap** — the visible surface of D-011's spatial confidence budget. A toggleable rendering mode showing where the model is survey-grade, estimated, stale, or unverified. Trust as a visible product feature, not buried in a spec sheet.

**3. Constraint Solver** — physically and operationally valid layouts. Hard rules (capacity max, aisle widths, exit clearance, fire egress), heritage rules (listed-building protections, load-bearing constraints), accessibility rules (wheelchair paths, accessible WC distance, hearing loop coverage), and soft rules (sightlines, acoustic comfort, table-to-table proxemics). Required substrate for capability 4.

**4. Prompt-To-Perfect Event** — natural language description in, three viable layout options out. "180-person black-tie dinner, dance floor, speeches, bar, wheelchair access, £18k budget" produces three layouts with table arrangements, flow, capacity, cost breakdown, ops notes, and reasoning. The demo-winning capability. Depends on capability 3 + capability 6 + capability 7.

**5. Live Planning Room** — multiplayer planning with in-3D-space pinned comments, version history, approvals, audit trail. Already partially captured in CRDT collaboration ADR work. Pinned-comments-in-3D is the additional surface that makes it Figma-for-venues.

**6. Pricing Engine** — every layout compiles into a priced proposal. Venue hire, catering per cover, AV rental, staffing, vendor markups, contingency. The proposal becomes a quote, the quote becomes a contract. Turns Venviewer from planning tool into transaction broker.

**7. Lookbook / Style System** — each venue carries a documented aesthetic (palette, lighting moods, decor traditions, vendor aesthetics). Prompt-to-layout output respects venue style. "Black tie dinner" produces different visual results in Trades Hall vs. a modern hotel atrium because the aesthetic context differs.

**8. Event Ops Compiler** — the moat. Every layout compiles into a Banquet Event Order (BEO), staff checklist, furniture pick list, load-in sequence, vendor requirements, accessibility notes, fire egress markers. The bridge from planning to execution. None of the current market players (Cvent, Matterport, Prismm) close this gap. Venviewer that compiles to BEO is venue-indispensable.

**9. One-Click Cinematic Share** — every layout produces shareable assets: browser link with proper Open Graph cards, 15-second cloud-rendered cinematic video, still image pack, PDF proposal, before/after empty-room-vs-event comparison. The viral loop. Cinematic Mode (existing in roadmap as brief Spread VII) is the rendering substrate; this is the distribution surface.

**10. Audit Trail / Provenance Log** — every change to a layout, every approval, every comment, every version, logged immutably. Required for enterprise contracts. Not viral, not flashy, contract-enabling.

**11. Revenue Optimizer** — for venues, layouts that increase revenue without harming guest experience. More covers, better bar placement, upsell opportunities, staffing implications, margin estimate. Calibrated against explicit guest-experience thresholds (minimum table-to-table distance, dance floor area per guest, bar throughput per 100 attendees) so optimization cannot violate experience quality. Encodes the values constraint "guest experience over venue margin" as a hard floor.

### Strategic priority

If forced to rank for strategic importance:

1. Event Ops Compiler (capability 8) — the moat.
2. Prompt-To-Perfect Event (capability 4) — the demo.
3. Verified Photoreal Twin (capability 1) — already underway.
4. Truth Heatmap (capability 2) — already underway.
5. Constraint Solver (capability 3) — substrate.
6. One-Click Cinematic Share (capability 9) — viral growth.
7. Pricing Engine (capability 6) — commercial bridge.
8. Live Planning Room (capability 5) — sales process.
9. Lookbook / Style System (capability 7) — luxury differentiator.
10. Audit Trail (capability 10) — enterprise enabler.
11. Revenue Optimizer (capability 11) — power feature with values guardrails.

### Activation gates

Build order is constrained by foundation readiness, not by feature priority. Specifically:

- Audit findings T-080 through T-099 must close before any product-vision capability ships to customers, because capabilities built on a broken substrate are below the bar.
- Operational baseline (T-054 + T-060–T-072) must be in Tier 1/2 deploy before any capability handles real customer data, because shipping to customers without monitoring, backups, error tracking, and proper email is below the bar.
- Capabilities 1, 2, 3 are partially underway in existing architecture; full ship gated on T-091 (Trades Hall real evidence).
- Capabilities 4, 6, 8 form a tightly coupled cluster — prompt-to-layout generates costed BEO-compilable plans. Build them as one effort, not three.
- Capability 9 (Cinematic Share) is gated on capability 1 shipping (no point sharing layouts of procedural geometry).

## Consequences

Venviewer's public positioning is no longer "a venue planning tool." It is "the operating system for physical events at luxury venues." Marketing copy, brief surfaces, and investor materials should reflect this framing once the substrate supports it (after T-091 closes).

Engineering work is now organised around the eleven capabilities, not around platform-level features in isolation. Schema design, rendering pipeline decisions, and infrastructure choices should be evaluated against "does this serve one or more of the eleven capabilities."

The S+ tier standard (CLAUDE.md) means none of these capabilities ship until each is genuinely best-in-class. "Prompt-To-Perfect Event" that produces layouts violating capacity rules is below the bar — wait until the constraint solver is solid. "Cinematic Share" that produces a 5-second video instead of 15 is below the bar — wait until cloud rendering is solid.

## Alternatives considered

- **Build a generic 3D event planner** (Cvent / Prismm category): rejected. The market is served. Differentiation requires the loop, not just the geometry.
- **Build a venue marketing tool** (Matterport category): rejected. Tour viewing is commodity. The loop differentiates.
- **Build a wedding planning AI** (consumer category): rejected. Wrong customer (B2C vs B2B), wrong unit economics, wrong defensibility.
- **Build only capabilities 1–4 as MVP**: rejected as the long-term vision, accepted as the build sequence. The vision is all eleven; the first ship is the foundation.
