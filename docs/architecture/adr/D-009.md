# D-009 — Typed spatial-layer graph (VSIR-0)

Status: proposed
Date: 2026-04-25
Authors: Blake
Supersedes: none

## Context

D-003a reframes the photographic pipeline as a typed graph with interchangeable render recipes rather than a fixed set of five asset filenames. That reframing requires a formal interface specification: which node types exist, what each accepts as input, what each produces, and how implementations swap. Without an interface spec, "typed graph" is rhetoric. Standards are moving the same direction: OpenUSD v26.03 added `UsdVolParticleField3DGaussianSplat`, and Khronos KHR_gaussian_splatting reached release-candidate status in February 2026. VSIR (Venviewer Spatial Intermediate Representation) version 0 should mirror these where possible so that future-format adapters are translation, not rewriting.

## Decision

Define VSIR-0 with six canonical layer families:

1. **CaptureSources** — raw input artifacts: panoramas, fisheye stills, video frames, phone scans.
2. **GeometricSubstrate** — the geometry that everything else binds to: OBJ mesh, BakedSDF, point cloud, parametric model.
3. **PoseFrame** — canonical coordinate frame and per-capture extrinsics: COLMAP, MapAnything, VGGT.
4. **BaseAppearance** — the photometric base layer: projective texturing, neural texture, mesh-bound spherical Gaussians.
5. **ResidualAppearance** — view-dependent overlay: cropped 3DGS, 2DGS, Ref-Gaussian, baked neural residual.
6. **DeliveryFormat** — wire format and runtime container: glTF + KHR_gaussian_splatting, OpenUSD, internal binary.

Each family has a typed interface; multiple implementations may be registered per family. The VSIR graph for a venue picks one implementation per family at render time. USD-isomorphic where possible — when Khronos and OpenUSD ratify the splat spec, the VSIR layer mirrors their semantics so a future glTF/USD adapter is a translation, not a rewrite.

## Consequences

D-003a, D-004a, D-005a, D-006a all become concrete by referencing this spec. Future ADRs about specific implementations (e.g., "we tried BakedSDF as GeometricSubstrate") don't need to amend D-009 — they specify which family they populate. Schema work in code (the AssetVersion bundle, D-014) reflects typed family nodes, not asset filenames. Switching cost between renderer recipes drops from "rewrite pipeline" to "swap a node."

Falsifying experiment: implement one alternate backend (BakedSDF or SuGaR base) against the graph. If it requires touching unrelated code, the abstraction has leaked.

## Alternatives considered

- **No formal spec, leave the typed graph informal** — rejected. Without a spec, the abstraction leaks within months and future ADRs can't reference it cleanly.
- **A full LLVM-style multi-pass optimizer** — rejected. Premature for solo founder per Claude-DR Part 2.
- **Adopt OpenUSD or glTF native and skip VSIR** — rejected. Both standards are moving but neither is ratified for splats yet (Khronos RC Feb 2026, expected Q2 2026 ratification with ~25% slip risk per Claude-DR). VSIR-0 lets us ship while remaining isomorphic.
