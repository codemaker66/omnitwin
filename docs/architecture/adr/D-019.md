# D-019 — VSIR-0 Concrete Schema: Four-Source Synthesis

Status: accepted
Date: 2026-04-29
Authors: Blake
Supersedes: none
Depends on: D-009, D-011, D-014, D-015
Relates to: D-018 (product vision — the eleven capabilities all build on this substrate)

## Context

D-009 committed to the typed-spatial-layer-graph architecture as the canonical in-memory representation of a venue. D-011 added the Spatial Confidence Budget. D-014 added the Venue Artifact Factory. D-015 added Capture Certification Tiers. None of those ADRs specified the concrete schema, signing format, or migration plan that operationalizes the architecture in PostgreSQL, Drizzle, and the existing Venviewer codebase.

VSIR-0 is that operationalization. To produce a high-confidence specification, four independent deep research outputs were commissioned from different reasoning systems, given the same architectural brief. The four were reviewed in sequence and synthesized into a single committed design. Where all four converged, the consensus is the design. Where they diverged, the strongest single answer was selected with explicit reasoning. Where any source made a verifiable technical error, the error was identified and corrected.

This ADR captures the synthesis as the committed VSIR-0 design.

## Decision

### The four-source consensus

The following decisions were reached by all four sources independently. They form the load-bearing architectural foundation:

- Schema split into per-domain TypeScript files before any VSIR table is created
- SQL migrations starting at 0019 directly, no no-op refactor migration
- `vsir_*` table prefix in PostgreSQL (no schema-level namespace)
- Single `packages/api/src/db/schema/vsir.ts` location for the Drizzle table definitions
- Branded TypeScript IDs for compile-time entity-type distinction
- Zod discriminated unions for prim variants (Xform, Mesh, Splat, Pano, Volume, FixtureRef, Space)
- Application-layer tenant scoping; PostgreSQL RLS deferred with documented activation triggers
- One-way VSIR → USD export only; round-trip is v2+ ADR work
- `UsdVolParticleField3DGaussianSplat` for SplatPrim USD mapping (OpenUSD v26.03 schema)
- in-toto v1 Statement wrapped in DSSE envelope for signed bundles
- Static Ed25519 signing in v1 with KMS-held private keys; Sigstore-keyless signing deferred to v2
- "Header in DB, body in file" payload split — full prim arrays do NOT live in JSONB columns; they live in immutable content-addressed files referenced from the database

### Cross-source best-of selections

Where the four sources diverged, these specific decisions were selected for the final design with the reasoning noted.

**Status enum pattern with partial unique active indexes.**
Source: ChatGPT-1, ChatGPT-2.
Decision: Use entity-specific Postgres status ENUMs following a common lifecycle pattern (e.g. `vsir_capture_status`, `vsir_registration_status`, `vsir_artifact_status`), with partial unique indexes enforcing "exactly one active per logical key" where applicable. Do NOT force a single universal status enum across all nine entities — different entities have genuinely different lifecycles (a capture session doesn't "supersede"; an artifact does).
Reasoning: Database-enforced singleton-active is significantly more robust than application-layer enforcement. Per-entity enums preserve semantic accuracy without forcing awkward common vocabularies.

**JSONB shape, SHA256 format, and time-ordering CHECK constraints.**
Source: ChatGPT-1.
Decision: Adopt all three classes of CHECK constraints on relevant columns.
Reasoning: Cheap database-level invariants that prevent accidentally-stored arrays in object columns, malformed hash values, and out-of-order timestamps.

**Three-migration split.**
Source: ChatGPT-2.
Decision:

- `0019_vsir_core_tables.sql` (CREATE TABLE statements + ENUM types)
- `0020_vsir_indexes.sql` (CREATE INDEX statements)
- `0021_vsir_comments_and_checks.sql` (COMMENT ON, additional CHECK constraints, refinements)

Reasoning: Indexes can iterate without rewriting table-creation migrations. Comments and refinements can iterate without table churn.

**Branded TypeScript IDs.**
Source: Claude.
Decision: Adopt `Brand<string, "CaptureSessionId">` pattern for every entity ID.
Reasoning: Compiler catches `captureSessionId` being passed where `runtimePackageId` is expected. Standard pattern in serious TypeScript codebases (Stripe, Linear).

**Matrix4d-only transform encoding.**
Source: Claude.
Decision: 16-element column-major array of doubles. No TRS discriminated union variant. No JSONB transformation matrices.
Reasoning: One canonical form is reasoned about more easily than two. JSONB for fixed-shape numeric data is wasteful and unindexable.

**Narrow single-subject QA certificate model.**
Source: Claude.
Decision: A QA certificate attests to one (capture_session, optional spatial_layer_version, optional semantic_graph_version) tuple. Not multi-subject across runtime packages and asset versions.
Reasoning: A certificate that can attest to "any of N subject types" is harder to reason about than separate certificates per subject scope. Avoids the over-engineered conditional CHECK constraint pattern.

**BRIN indexes on append-heavy `created_at` columns.**
Source: ChatGPT-2.
Decision: Add BRIN indexes on `created_at` for `vsir_capture_sessions`, `vsir_asset_versions`, `vsir_runtime_packages`.
Reasoning: BRIN indexes are roughly 1000× smaller than B-tree for the same column. For tables that grow append-only over years, BRIN gives fast time-range queries at almost zero index storage cost.

**Controlled polymorphism in exactly two places.**
Source: ChatGPT-2.
Decision: Proper FKs everywhere except `vsir_asset_versions.subject_id` and `vsir_access_policy_versions.scope_id`. These two columns use `(subject_type, subject_id)` and `(scope, scope_id)` polymorphic patterns with enum-typed discriminators. Application-layer validation enforces that polymorphic IDs resolve to real entities.
Reasoning: Storage subjects (any of 9 entity types) and policy scopes (venue/space/capture_session/runtime_package) are genuinely polymorphic. Forcing them into FK shape requires ugly join tables. Limiting polymorphism to exactly two well-defined columns is the clean compromise.

**`parent_layer_version_id` self-referencing FK on spatial_layer_versions.**
Source: ChatGPT-2.
Decision: Add a nullable self-referencing FK to enable processing lineage chains (raw_scan → cleaned_mesh → gaussian_splat → hybrid).
Reasoning: Provenance through multi-stage processing is a first-class concern for VSIR-0. Self-referencing FK makes it queryable.

**`not_before` / `not_after` columns on `vsir_runtime_packages`.**
Source: ChatGPT-2.
Decision: Add timezone-aware time-window validity columns.
Reasoning: Enables staged rollouts (sign now, publish later), preview links with expiry, and time-limited validity for contractual delivery.

**`coordinateSystemSchema` enum.**
Source: ChatGPT-2.
Decision: Make coordinate-system tracking first-class via enum: `rhs_y_up_meters`, `rhs_z_up_meters`, `matterport_local`, `e57_local`, `usd_rhs_y_up`, `usd_rhs_z_up`.
Reasoning: VSIR has to bridge Three.js (Y-up RH), USD (configurable), Matterport (its own), and E57 (its own). Tracking coordinate-system via enum eliminates "what space is this matrix in" guesswork.

**Application-layer tenant guard returns 404, not 403.**
Source: Gemini.
Decision: When a request attempts to access another tenant's resource, return `404 not_found` rather than `403 forbidden`.
Reasoning: 403 leaks information ("a resource exists at this UUID, you can't see it"). 404 is indistinguishable from "this UUID doesn't exist." Prevents UUID enumeration attacks.

### Signing flow

**Signing format.** in-toto v1 Statement wrapped in DSSE envelope.

**Detached attestation.** The DSSE envelope lives as a separate file in the bundle (`attestations/runtime-package.dsse.json`), NOT embedded in `manifest.json`.

Reasoning: Embedding the attestation inside the manifest creates a circular digest problem. Either the manifest hash is taken before the attestation is added (in which case the hash no longer matches the manifest as stored), or a "manifest excluding attestation" canonical-form rule must be defined and followed identically by every implementation. Detached attestation eliminates both problems.

**No JCS canonicalization required.**

DSSE PAE does not canonicalize JSON. It removes the need for JSON canonicalization by signing the exact serialized payload bytes together with the payload type. VSIR-0 explicitly does NOT require RFC 8785 / JCS in v1 unless a separate non-signature need for semantic JSON normalization arises (none currently exists).

Reasoning: DSSE's design goal is precisely to avoid security depending on canonicalization. Adding JCS adds an extra dependency, an extra failure mode (Node JCS implementation differs from Python JCS implementation by one character), and no security benefit. Two of the four deep research sources reached for JCS reflexively; this was identified as a technical error and corrected.

**Signing flow (10 steps, verified):**

1. Build deterministic file list sorted by path
2. Hash every file's raw bytes with SHA-256
3. Serialize `manifest.json` once (whatever the JSON serializer produces is what gets hashed)
4. Hash `manifest.json` raw bytes
5. Build in-toto Statement over manifest hash + bundle subject hashes
6. Serialize Statement to UTF-8 JSON bytes
7. DSSE-sign exact Statement bytes with Ed25519 from KMS
8. Store the DSSE envelope as a separate attestation file (`attestations/runtime-package.dsse.json`)
9. Store signature metadata on RuntimePackage row (signature, signature_type, public_key_fingerprint, signed_at, nonce)
10. Parse the DSSE envelope minimally to extract payload and signatures during verification. Verify the DSSE signature over the exact payload bytes BEFORE trusting or semantically parsing the in-toto Statement and predicate.

**Verification flow (15 steps):** documented in the full spec; mirrors the signing flow in reverse with explicit nonce uniqueness, time-window, and version-tuple-not-already-published checks.

### Standards posture

**OpenUSD v26.03 `UsdVolParticleField3DGaussianSplat`:** stable schema, used as primary USD export target.

**Khronos KHR_gaussian_splatting:** alignment target only, NOT stable dependency. Khronos officially announced `KHR_gaussian_splatting` as a Release Candidate on February 3, 2026, explicitly inviting industry feedback before ratification. The current public spec status is still Release Candidate. Final ratification was originally targeted for Q2 2026 (the current quarter) but as of this ADR's authorship has not been confirmed. Production deployments by CesiumJS, Esri ArcGIS, Scaniverse, and the XGRIDS ecosystem are already shipping against the Release Candidate.

VSIR-0 implements against the Release Candidate now. Public copy and customer-facing materials must describe alignment as "Release Candidate aligned" or "emerging standard aligned," not "ratified standard." When ratification lands, copy can be updated in one place and the messaging upgrades automatically. The SPZ-handling layer in the runtime will be built with a thin abstraction so any spec adjustments between RC and final ratification are contained to that layer.

### Gaps requiring separate follow-up ADRs

All four deep research sources independently failed to address two architectural concerns. Both are committed as separate future ADRs:

**D-020: VSIR-0 QA jobs orchestration queue.**
All four sources describe validation passes (splat_dedup, mesh_hole_for_redaction, lod_split, pano_seam_blend, confidence_voxelize, ifc_validate, mvr_resolve, attest) as worker processes triggered by job rows, but none define the queue table. As written, the validation passes cannot actually be triggered. D-020 designs the small `vsir_qa_jobs` table (approximately 6-8 columns: id, pass_name, subject_id, subject_type, status, attempt_count, error, timestamps) and the worker dispatch protocol.

**D-021: VSIR-0 partition strategy for long-lived growth.**
For a venue captured weekly over years, `vsir_spatial_layer_versions` and `vsir_qa_certificates` will grow into the millions of rows. None of the four sources addressed PostgreSQL declarative partitioning. Acceptable to defer for v0.1 (Trades Hall is one venue, growth is slow), but D-021 documents the trigger conditions for activating partitioning and the recommended scheme: likely HASH or LIST partitioning by `venue_id` (group all rows per venue together), or RANGE partitioning by `created_at` month or year (group by time window), depending on query patterns and row volume. Range partitioning on `venue_id` directly is NOT applicable because UUIDs do not have meaningful range ordering.

### Note: supersession of earlier draft approaches

Two of the four input deep research sources (Claude's spec, Gemini's spec) included a `prims jsonb` column directly on `vsir_spatial_layer_versions`, storing the full prim array as queryable JSONB in the database. D-019 supersedes that approach in favour of the "header in DB, body in file" pattern adopted by ChatGPT-1 and ChatGPT-2: large prim arrays live in immutable content-addressed asset files referenced via `vsir_asset_versions`, while the database row stores headers, hashes, version metadata, summary statistics (`prim_count`, `splat_count`, etc.), and storage pointers.

This is a material design change. Implementers reading both D-019 and the archived deep research outputs MUST follow D-019, not the earlier drafts. The archived sources are kept for reasoning-record purposes, not as implementation guidance.

### Implementation effort estimate

VSIR-0 schema implementation: roughly 3-4 weeks of focused work.

- Schema split refactor: 1 day
- Three-migration sequence: 1 week
- Drizzle table definitions in `packages/api/src/db/schema/vsir.ts`: 2 days
- Zod schemas in `packages/vsir/src/*.ts`: 3 days
- Validation pass worker scaffolding (excluding actual pass implementations): 3 days
- Application-layer guards in `packages/api/src/auth/vsir-access.ts`: 2 days
- Signing flow with KMS integration: 3 days
- Verification flow at backend ingestion: 2 days
- Test coverage: 3-5 days

Per-pass implementation (the actual splat_dedup, mesh_hole_for_redaction, lod_split, pano_seam_blend, confidence_voxelize, ifc_validate, mvr_resolve, attest workers): additional 2-3 weeks spread across multiple workers.

Total: 6-7 weeks of focused work to land the full thing.

VSIR-0 implementation does NOT begin until:

1. Audit findings T-080 through T-099 close (current acquisition-readiness gap closure)
2. Operational baseline (T-054 Geass + T-060 through T-072) is in place
3. Trades Hall first training run completes (T-091)

Per the S+ tier standard in CLAUDE.md, building product features on a substrate that doesn't yet match its public claims is below the bar. VSIR-0 is the substrate the eleven product capabilities in D-018 will build on. It must be implemented correctly, with the foundation work closed first.

## Consequences

VSIR-0 is now committed as a defined design. Future engineering work on the typed-spatial-layer-graph references this ADR as the canonical specification. Subsequent ADRs (D-020 qa_jobs, D-021 partitioning) refine it without rewriting it. The four deep research outputs are archived for posterity.

The detached-attestation decision and no-JCS decision both correct technical errors found in two of the four input sources. Future implementation must follow the corrected design, not any single source verbatim.

The supersession note in "earlier draft approaches" is binding: D-019's "header in DB, body in file" pattern overrides the `prims jsonb` approach in two of the four input sources.

The KHR-as-RC-alignment-target wording must propagate to all public copy. Specifically: the brief at `apps/web/public/private/brief/trades-hall-2026-04-27/` contains language that should be reviewed against this wording before the next external presentation.

## Alternatives considered

- **Adopt one source verbatim.** Rejected. Each of the four has at least one weakness the others avoid.
- **Defer the synthesis until all foundation work closes.** Rejected. Synthesis is most accurate while all four sources are fresh in context. Capturing now and gating implementation on foundation closure is cleaner than deferring both.
- **Skip controlled polymorphism, use proper FKs everywhere.** Rejected. Forces ugly join tables for genuinely polymorphic storage subjects and policy scopes.
- **Use JCS canonicalization "to be safe."** Rejected. DSSE PAE eliminates the need; adding JCS adds dependencies and failure modes for no security benefit.
- **Embed attestation in `manifest.json`.** Rejected. Creates circular digest problem.
- **Store full prim arrays in JSONB on `vsir_spatial_layer_versions`.** Rejected. Acceptable for tens of prims, problematic at hundreds of thousands. Header-in-DB-body-in-file scales better and keeps database row sizes lean.
- **Single universal status enum across all nine entities.** Rejected. Forces awkward semantics where lifecycle stages don't actually apply (a capture session does not "supersede").
