# D-008 — Venue is the tenant unit

Status: Accepted
Date: 2026-04-23
Authors: Blake
Supersedes: Path X / Path Y / Path Y+ multi-tenancy proposals from conversations prior to reading the actual codebase (2026-04-23 clarification).

## Context

The existing codebase's multi-tenant architecture uses `venues` as the unit of tenancy. Every user-data table has a `venue_id` foreign key; routes scope access via `canManageVenue()` / `canAccessResource()`. No separate `Tenant` entity exists. The architecture (confirmed by reading all route files 2026-04-23) is already venue-scoped multi-tenant; the README's "single-tenant" wording is misleading, not an architectural limitation. Sub-tenant hierarchy (Aman Group → Aman Venice → Aman Tokyo) and cross-venue subcontractor access (Trades Hall invites Regis Catering) are foreseeable but not required for v1.

## Decision

Venues ARE the tenant unit. No separate `Tenant` entity, and none required for v1.

## Consequences

- README must be updated to remove "single-tenant" framing (tracked: separate commit, future work).
- No schema refactor required for multi-tenancy; the architecture is sound.
- Future multi-property chains (Aman with multiple venues) are modelled by one Aman user having staff/hallkeeper role on multiple venues; when a true hierarchy is needed, add `Tenant` table above `Venue` then.
- Subcontractor cross-venue access is a future `SubcontractorAccess` table, not a tenant-layer concern.

## Alternatives considered

- **Separate `Tenant` entity above `Venue`** — rejected for v1. Over-engineering; venues ARE the tenant unit in venue-planning software.
- **Path X / Path Y / Path Y+ multi-tenancy proposals** — rejected. Predated reading the actual codebase; the existing architecture is already venue-scoped multi-tenant, so these alternative path designs were redundant.
