# Research Brief R5 — Venviewer (omnitwin2): Timeline Rendering & Real-Time Collaboration Architecture

## TL;DR
- **Build the timeline DOM-first, not canvas.** At Venviewer's stated scale (dozens of lanes, hundreds-to-low-thousands of visible blocks), a virtualised DOM timeline hits 60fps comfortably; canvas is premature optimisation that would cost you accessibility, text quality, and print for no benefit you can perceive. Use TanStack Virtual (two-axis windowing), dnd-kit or react-aria for accessible drag, and a server-side Puppeteer "render-all" print path.
- **Make the server authoritative for booking state — never CRDT-merge bookings.** CRDTs merge concurrent writes, which is exactly wrong for a double-booking invariant (both bookings survive the merge). Use a server-authoritative command/validation model with optimistic UI + rollback for booking mutations, and reserve CRDTs (Yjs) only for free-text event-planning notes.
- **Recommended realtime stack: a single authoritative WebSocket service (your own Node `ws`/uWebSockets.js, or Cloudflare Durable Objects via PartyServer) that validates every booking mutation against conflict rules before broadcast, plus lightweight presence/soft-locks over the same channel, plus optional Yjs for notes.** This keeps operational complexity low for a solo/small team while guaranteeing correctness.

## Key Findings

### Part 1 — Timeline rendering

1. **Every serious commercial scheduler is virtualised DOM, not canvas.** Bryntum Scheduler (the performance leader) uses "virtual rendering, minimal DOM footprint, extensive element reuse" — buffered rendering with recycled DOM built on their Grid engine. FullCalendar migrated its internals to a Preact virtual DOM and is adding true virtual rendering for resources/events across v7.1–v8. Syncfusion loads timeline resources/events "virtually on every scroll action." vis-timeline is DOM+SVG (canvas only for some drawing). Only niche, extreme-scale operational products (e.g. ScheduleJS/FlexGanttFX) use canvas — and they target tens of thousands of live items, far above Venviewer's scale.

2. **Published benchmark proves DOM is fast enough by a wide margin.** Bryntum's own cross-vendor benchmark uses a dataset of **2,500 resources and 50,000 events (20 per resource)**, all schedulers sized 1024×768 with grouping/sorting turned off — an order of magnitude beyond Venviewer's target. Bryntum records 291ms small-render / 666ms large-render and 48–60fps scroll, and Bryntum/ag-Grid/ExtJS grids average "around 60 frames per second." That is enterprise-Gantt scale done in pure DOM. At Venviewer's dozens-of-lanes, low-thousands-of-blocks scale, a well-built virtualised DOM timeline will not be the bottleneck. (Caveat: these are vendor-measured numbers on a 2016 MacBook Pro — directional, not guaranteed.)

3. **TanStack Virtual is the right virtualiser; react-window/react-virtualized are legacy.** TanStack Virtual is headless, framework-agnostic, and heavily maintained — `@tanstack/react-virtual` is at v3.14.5 with **13,662,421 weekly downloads on npm** (TanStack's own site reports 19,620,208 weekly and 810.6M total across the Virtual family, calling it "the most popular and most used virtualization engine for modern web apps"). It explicitly supports combining a vertical + horizontal axis for grid/lane virtualisation — exactly the two-dimensional (lanes × time) windowing Venviewer needs. It uses transform-based absolute positioning (`transform: translateY(...)`). Watch one React 19 gotcha: the compiler/memoisation can hoist `getVirtualItems()` so it returns 0 items; the community workaround is holding the virtualizer in a ref, and setting `useFlushSync: false` removes a React 19 flushSync warning during scroll.

4. **CSS `content-visibility: auto` is a strong complement, not a replacement, at this scale.** Google's web.dev documents a demo going "from a 232ms rendering time to a 30ms rendering time — a 7x performance boost," and real-site tests show ~40% reductions (e.g. Erwin Hofman measured 41.2% on a Windows 10 Toshiba laptop). Nolan Lawson used it (Sept 2024) to fix a Fediverse emoji picker with "19k custom emojis" where the page "freezes for like a full second" — roughly 40k DOM elements, far above Lighthouse's ~1,400 recommendation — without touching ARIA or find-in-page. But it loads all data into the DOM (memory cost at thousands of nodes) and needs `contain-intrinsic-size`. For Venviewer, use true virtualisation for the block layer and `content-visibility`/`contain: layout paint` on off-screen lane groups. Apply `will-change: transform` only on the element actively being dragged (never leave it on permanently — it forces wasteful GPU layers).

5. **Canvas is genuinely unnecessary here — and Figma's own story explains why.** Figma went canvas/WebGL because they needed masking, blurring, angular gradients, zoom-not-scroll geometry, and millions of objects on an infinite canvas — none of which apply to a lane/time grid of a few thousand rectangles. Canvas costs you: text rendering quality (Figma had to build its own text-layout engine), accessibility (screen readers see "a black hole"), hit-testing complexity, and print/export. The new experimental HTML-in-Canvas API (Chrome origin trial, 2026) is trying to give canvas apps back DOM accessibility — a strong signal that going canvas means giving up things you'd rather keep. If you ever need a hybrid, the pattern is canvas for the block layer + DOM overlay for interactive handles/popovers (react-konva even ships an `<Html>` portal for this), but you should not need it.

6. **Accessible drag is the priority area — and it's where even the best commercial tools fail.** Per the W3C ARIA Authoring Practices Guide, there is **no** dedicated timeline/scheduler/calendar pattern — the closest applicable model is the **Grid pattern**: arrow keys move focus between cells ("Moves focus one cell to the right… If focus is on the right-most cell in the row, focus does not move"), `Home`/`End` to row ends, `Ctrl+Home`/`Ctrl+End` to grid corners, `Page Up`/`Page Down` by an author-determined number of rows, `Enter`/`F2` to enter cell editing, `Escape` to restore navigation, and `Shift`/`Ctrl`+arrows for selection. Critically, APG patterns are **informative, not normative** — WCAG 2.1.1 (Keyboard, Level A) only requires the operation be keyboard-operable at all, with "no normative requirements… to support specific keystrokes." **FullCalendar does not support keyboard drag/move/resize** — events became focusable/tabbable in v5.10 (Nov 2021), but moving them is mouse or touch-long-press only; keyboard drag has been an open, unshipped feature request (GitHub issue #2535) since 2015, still tagged for a future milestone. **Bryntum** supports keyboard *navigation* between events (inherited from its Grid) but its own staff state it has "no/very limited support for keyMap keyboard shortcuts," and there is **no native keyboard-driven drag/move/resize** — you implement it yourself. This means Venviewer can *exceed* the commercial benchmarks simply by shipping real keyboard DnD.

7. **For accessible DnD, choose react-aria or dnd-kit — never native HTML5 DnD.** Native HTML5 drag-and-drop supports neither keyboard nor touch dragging. **react-aria** (Adobe) ships a from-scratch keyboard/screen-reader DnD model: press `Enter` to pick up, `Tab` between drop targets, `Enter` to drop, `Escape` to cancel, with built-in screen-reader announcements and "full parity" across mouse/touch/keyboard. **dnd-kit** is deliberately not built on HTML5 DnD, ships a `KeyboardSensor` (Space/Enter to lift, arrows to move, Escape to cancel) with a customisable `announcements` API and an off-screen live region, and supports custom coordinate-getters so arrow keys map to time-slot/lane movements. Both auto-manage `aria-roledescription`, focus restoration, and live-region announcements. For Venviewer's "pick up block → arrow to move by slot → modifier+arrow to resize → Enter to drop" with announcements like "Booking moved to Main Hall, Tuesday 14:00–18:00," a custom coordinate-getter over dnd-kit, or react-aria's collection DnD, is the way. dnd-kit's docs also stress using *position*-based announcements ("moved to position 2 of 5") rather than zero-based indices.

8. **React 18/19 concurrency helps scrubbing/zoom.** Wrap non-urgent timeline recomputation (zoom-level change, filter, re-layout) in `useTransition`/`startTransition` so the input gesture stays high-priority and never blocks; use `isPending` for a subtle pending state. Do **not** wrap the drag position update itself (it's urgent), and don't treat transitions as a fix for genuinely expensive renders — reduce the work first. React Compiler reduces the need for manual `useMemo`/`useCallback` in memoisation-heavy timeline code, but verify virtualizer output isn't optimised away (the ref workaround above).

9. **Print/PDF: virtualisation breaks printing, so build a separate "render-all" path.** This is a documented failure — Syncfusion confirms that with virtualisation enabled, "only the virtual rows rendered in the DOM will be printed." The robust pattern for Venviewer's Day Sheet is server-side headless Chromium (Puppeteer/Playwright) rendering a dedicated print route that disables virtualisation (renders all lanes/blocks for the chosen day), uses `@page` landscape CSS (A4/A3), `preferCSSPageSize: true`, `printBackground: true`, and `break-inside: avoid` on blocks. Puppeteer gives pixel-faithful output reusing your React layout. `@react-pdf/renderer` produces beautiful PDFs but forces you to rebuild the layout in its own component model (data-driven re-layout) — better only if you want a purpose-designed day-sheet document rather than a faithful rendering of the timeline. SVG export is a viable intermediate for a crisp vector day sheet.

### Part 2 — Real-time collaboration

10. **CRDTs are the wrong tool for booking state — this is well-established.** CRDTs guarantee convergence but explicitly **cannot enforce global invariants** like uniqueness or "no double-booking." Loro's "When Not to Use CRDTs" documentation uses the exact venue case verbatim: two clients `book("A101@2pm", {by:"A"})` and `{by:"B"}` concurrently, and "after sync: both entries are present → invariant violated without coordination," recommending instead "transactional authority (central server or database)" or "hybrid: CRDT for UI drafts, authoritative booking via server." Academic work (LoRe, arXiv 2304.07133) shows the same for a calendar app where two offline devices book overlapping vacation and the merge violates the invariant. CRDTs are for character-level text merge (event notes), not business records with invariants.

11. **Liveblocks: excellent DX, managed-only, per-MAU pricing, not for authoritative data.** Presence, Storage (LiveObject/LiveList/LiveMap), and a managed Yjs backend with dashboard/webhooks/REST. Pricing is credit-based on top of a free tier: **Pro comes with $30 monthly credits, Team with $600 monthly credits**, Enterprise custom; billing is per-MAU where "one unique userId corresponds to one user" regardless of session length, and free-tier projects "will be paused when they exceed the included free tier usage" (free storage 256MB; Pro 8GB). It open-sourced its sync-engine core + dev server (AGPL v3) in 2025 for local dev, but production self-hosting is explicitly not yet packaged ("a production-ready self-hosted deployment… requires additional infrastructure… not yet easy to package"). Lock-in risk is real; per-room pricing can spike with dynamic rooms. Good for presence/notes; **not** a source of truth for financial booking state.

12. **PartyKit is now Cloudflare (acquired April 2024) and lives on as PartyServer/Durable Objects.** Existing PartyKit works; the forward path is the open-source `partyserver` library deploying to your own Cloudflare account (Durable Objects, pay-per-use, free tier). A Durable Object is a single-threaded stateful actor — an ideal serialisation point for authoritative per-venue (or per-day) booking logic: one DO owns a room, validates mutations, and broadcasts. Y-PartyServer adds Yjs if you want it. Strong fit for server-authoritative logic *in the room* with UK/EU edge presence.

13. **Supabase Realtime: three modes, and postgres_changes does not scale.** Postgres Changes (CDC via logical replication) is single-threaded to preserve order — Supabase confirms "compute upgrades don't have a large effect on the performance of Postgres change subscriptions" — with ~50–200ms latency (WAL processing). Broadcast is direct/server-mediated at "typically under 50ms," and Supabase now explicitly says "we recommend using Broadcast for most use cases," including the newer **Broadcast-from-database** pattern (a trigger + `realtime.broadcast_changes()`). Presence is CRDT-based/eventually consistent — fine for "who's viewing," wrong for authoritative counts. Built on Elixir Phoenix; RLS is enforced on channel join. Fit is strong **if Postgres is your source of truth**: write the booking through a transactional API (conflict check in the DB), then broadcast-from-database the confirmed change.

14. **The 2025–26 sync-engine wave is mostly local-first/optimistic; a few fit correctness-first.** ElectricSQL (Postgres→SQLite shapes, last-write-wins by default), Convex (server-first reactive DB, GA — not a sync engine; reads/writes are server-authoritative), Zero by Rocicorp (reached 1.0 in June 2026; server-authoritative — "server accepts or rejects mutations" — with custom mutators and a read-only Postgres cache), Jazz (batteries-included local-first, CRDT-ish), PowerSync. Of these, **Convex and Zero are the server-authoritative ones** and could in principle enforce booking invariants; ElectricSQL/Jazz default to merge semantics wrong for bookings. All add a large platform commitment versus a focused WebSocket service.

## Details

### Recommended timeline rendering architecture (verdict: DOM-first)

**Stack:**
- **Virtualiser:** TanStack Virtual, two virtualizers composed — a vertical one over lanes (rooms/resources) and a horizontal one over the time axis. Render only visible lane×time-window blocks. Position blocks absolutely + `transform` inside a relatively-positioned spacer sized to `getTotalSize()`.
- **Block layer:** plain DOM (divs) — free text rendering (subpixel AA, HiDPI), focus rings, ARIA, find-in-page, and print. At low-thousands of blocks with virtualisation you'll keep live DOM node count in the low hundreds.
- **DnD + a11y:** dnd-kit with a custom `KeyboardSensor` coordinate-getter (arrows = move by snap interval/lane; modifier+arrows = resize; Enter/Space = pick up/drop; Escape = cancel) and a customised `announcements` live region; OR react-aria's collection DnD for Adobe's tested keyboard/SR model out of the box. Use a ghost/preview overlay (dnd-kit `DragOverlay` / react-aria `DragPreview`) removed from document flow for smooth drag.
- **Concurrency:** `useTransition` for zoom-level changes and filtering; keep the drag/scrub gesture on the urgent path.
- **CSS:** `contain: layout paint` on lane containers; `content-visibility: auto` + `contain-intrinsic-size` on collapsed/off-screen lane groups; `will-change: transform` only on the actively dragged block.
- **Interaction patterns to replicate from the commercial tools:** drag-to-create (double-tap/drag on empty space, per vis-timeline), drag-to-move and resize handles, ghost/preview during drag, snapping (custom snap intervals, per vis-timeline's custom snapping), a current-time indicator, multi-level zoom (ms→year, up to 3 header rows, per Bryntum), event constraints (lock a booking to its resource/time window, per FullCalendar `eventConstraint`), and a fixed left resource column with locked/pinned scrolling (the case Bryntum/ag-Grid optimise heavily). Prefer buffered/infinite scroll on the time axis with paged day/week navigation.

**Performance budget:** target 60fps (16.7ms/frame) during drag; keep live DOM block count in the low hundreds via virtualisation; initial render of a day view well under ~300ms (Bryntum does 291ms at ~100× the data). If you measure sustained jank at Venviewer's scale, the first moves are tightening virtualisation/overscan, memoisation, and `content-visibility` — not switching to canvas.

**When canvas would actually become necessary:** only if requirements change to tens of thousands of simultaneously-visible blocks, continuous zoom with re-tessellation, or heavy per-block visual effects (blur/masking/gradients). None are in scope. **Verdict: DOM-first, no canvas.**

### Recommended realtime architecture (verdict: server-authoritative)

**Core principle:** bookings are business-critical financial records; correctness beats latency; conflicts must be *prevented*, not merged.

**Recommended design — authoritative command model over WebSockets:**
1. **Single authoritative writer per venue/day "room."** Every booking mutation is a *command* (create/move/resize/confirm) sent to the server, not a direct state write. The server validates the command against conflict rules (overlap detection per lane/time) inside a transaction against the source-of-truth database, then either commits + broadcasts the confirmed new state, or rejects.
2. **Optimistic UI with reconciliation/rollback.** The client applies the change locally for instant feedback, tagged pending; on server ACK it's confirmed, on NACK it rolls back and shows "that slot was just taken." This is exactly how the double-booking invariant stays correct: the server is the single arbiter, and the loser of a race is rolled back rather than silently merged.
3. **Transport choice (pick by hosting preference):**
   - **Cloudflare Durable Objects via PartyServer** — a DO per venue/day is a single-threaded actor that naturally serialises mutations, holds soft-lock/presence state, and broadcasts; pay-per-use, EU edge, minimal ops. Best if you want managed, low-ops, edge locality.
   - **Your own Node service (`ws`/uWebSockets.js, or Socket.IO for convenience)** on a UK/EU host, with Postgres as source of truth — most control, no vendor lock-in, event-sourced/command-log friendly for the audit trail. Elixir Phoenix Channels/LiveView is the reference-quality alternative if the team knows Elixir.
   - **Supabase** — viable if you adopt Postgres-as-truth: do the conflict check in a transactional RPC, then **broadcast-from-database** the confirmed change (avoid postgres_changes at scale; avoid Presence for authoritative counts).
4. **Presence & soft-locking:** lightweight pub/sub on the same channel — publish "user X viewing/editing day D / block B." Soft-lock is advisory (warn "Alice is editing this booking"), never the correctness mechanism; the server-side conflict check is the real guarantee. Presence can be eventually-consistent (CRDT-style) since it's non-authoritative.
5. **Collaborative notes:** for Google-Docs-style event-planning notes, use **Yjs** (via Hocuspocus self-hosted, y-partyserver, or Liveblocks Yjs managed) **scoped strictly to the notes field** — never to booking state. This is the one place CRDT merge semantics are correct.
6. **Audit trail:** because the server validates and commits every mutation, persist an append-only command/event log (event-sourcing-lite) — this gives you the audit trail and server authority the domain requires.

**Operational-complexity verdict for a solo/small team:** the lowest-ops correctness-first option is **Cloudflare Durable Objects (PartyServer) for the authoritative room + presence, Postgres for durable storage/audit, optional Yjs for notes.** If you prefer full control and no lock-in, a single small Node `ws`/uWebSockets.js service in front of Postgres achieves the same with more DevOps. Avoid making any CRDT/local-first sync engine (Yjs, ElectricSQL, Jazz) the authority for booking state. Zero or Convex could serve as an all-in-one server-authoritative backend, but each is a large platform commitment.

**UK/Europe hosting:** Cloudflare (Durable Objects) and Supabase both offer EU regions; a self-hosted Node service can run in any UK/EU provider. All keep data resident in-region for GDPR.

## Recommendations

**Stage 1 — Prove the render path (build first).** Implement the virtualised DOM timeline with TanStack Virtual (two-axis), plain DOM blocks, drag-to-move/resize with dnd-kit including the keyboard sensor + live-region announcements from day one (your differentiator over Bryntum/FullCalendar). Benchmark against a synthetic worst case (e.g. 40 lanes × 2,000 blocks). *Threshold to change approach:* only if you cannot hold 60fps during drag after tuning overscan, memoisation, and `content-visibility` — then evaluate a canvas block layer with DOM overlay.

**Stage 2 — Server-authoritative bookings.** Model mutations as validated commands; implement conflict detection in a DB transaction; add optimistic UI with rollback. Start with a single Node `ws`/uWebSockets.js service or a Cloudflare Durable Object per venue/day. *Threshold:* if you outgrow single-writer throughput per room (unlikely at venue scale), shard rooms by day/space.

**Stage 3 — Presence, soft-locks, notes.** Add advisory presence + soft-lock UX over the same channel. Add Yjs-backed collaborative notes only for the free-text planning field.

**Stage 4 — Day Sheet print.** Build a dedicated server-side print route (virtualisation disabled, render-all) rendered by Puppeteer to landscape A4/A3 PDF; treat it as an operational document, not a screenshot.

**What would change these recommendations:** a requirement for tens of thousands of simultaneously visible blocks → reconsider canvas/WebGL; a requirement for genuinely offline-first multi-device editing of bookings → you'd need a coordination layer (still not naive CRDT), e.g. reservation tokens or per-slot ownership; a decision to standardise on Postgres-as-truth with modest realtime needs → Supabase broadcast-from-database becomes the simplest path.

## Caveats
- **Benchmark caveat:** Bryntum's published numbers are the vendor's own, measured on a 2016 MacBook Pro; treat them as directional evidence that DOM scales, not guaranteed numbers. Run your own benchmarks on target hardware.
- **FullCalendar roadmap is forward-looking:** virtual rendering for resources/events and infinite scroll are on the v7.1–v8 roadmap (2025 quarters) and described in future tense; verify shipped status before relying on it. Keyboard drag remains an open, unbuilt feature request.
- **Liveblocks self-hosting** is dev/test only today (AGPL core); do not assume production self-hosting. Pricing is credit-based per-MAU and can be unpredictable with dynamic rooms.
- **PartyKit** as a standalone brand is effectively wound into Cloudflare; build on `partyserver`/Durable Objects rather than legacy PartyKit Cloud for longevity.
- **HTML-in-Canvas API** is an experimental Chrome origin trial (2026), not a cross-browser production feature — cited only to illustrate the accessibility cost of canvas.
- Some secondary sources are vendor marketing or community blogs; the CRDT-correctness claims, ARIA patterns, and API behaviours are corroborated by primary docs (W3C APG, MDN, react-aria/dnd-kit docs, Supabase docs, Loro docs, Cloudflare/Liveblocks blogs).
