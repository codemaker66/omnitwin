# Calendar Sync for a Venue Booking System: Practical Pitfalls of Google Calendar API, Microsoft Graph, and ICS Feeds — and a P3 Architecture Verdict

## TL;DR
- **Ship the ICS read-only feed first as the "cheap early win" — but scope it honestly:** it is genuinely low-effort and good enough to display external "busy" blocks and soft-warn against conflicts, but Google re-fetches subscribed ICS feeds only every ~12–24 hours (Google's own Calendar Help wording has drifted from "every few hours" to "up to 8 hours" to "up to 12 hours"; Outlook/Microsoft 365 feeds refresh every ~1–4 hours, up to 24h worst case). It must never be the authoritative source that hard-blocks bookings. Treat ICS as advisory, not a booking gate.
- **For P3, build direct Google Calendar API + Microsoft Graph integrations using a "webhooks-as-hints + delta/sync-token reconciliation" pattern**, not webhooks alone. Both providers explicitly say notifications can be missed and carry no change payload; a periodic full/delta reconciliation (every ~15 min plus a daily sweep) is mandatory, not optional.
- **The highest-risk pitfalls for a UK venue are (1) DST "±1 hour" drift around the BST↔GMT transitions, (2) recurring-event edge cases silently corrupting availability, and (3) double-bookings from stale/dropped sync.** Store everything in UTC with an explicit IANA zone (Europe/London), expand recurrences into concrete instances yourself, and reconcile defensively. If team bandwidth is thin, a unified provider (Cronofy/Nylas) is a defensible buy for two-way sync.

## Key Findings

1. **Webhooks are hints, never a source of truth.** Google push notifications contain *no* change payload — just a "something changed" ping requiring a follow-up incremental sync. Channels expire (empirically ~7 days for Events, capped by Google) and there is no auto-renewal. Microsoft Graph subscriptions on calendar events max out at ~3 days (4,230 minutes) and must be renewed; Graph explicitly warns notifications can be dropped (e.g., under throttling) and provides "lifecycle notifications" specifically to tell you when you've missed some.
2. **Reconciliation via sync tokens (Google) / delta queries (Microsoft) is the backbone.** Both return a token/deltaLink you persist and replay; both invalidate the token (Google: HTTP 410 GONE; Microsoft: 4xx like `syncStateNotFound`) forcing a full resync. Google's 410 window has, in the past, dropped unexpectedly from ~weeks to hours without notice — you must handle 410 gracefully at any time.
3. **Recurring events are the number-one silent corruptor of availability.** Google and Microsoft model series differently (Google: series master + `singleEvents` expansion; Microsoft: `seriesMaster` + `calendarView`/`instances` endpoint). Modified instances become "exceptions"; cancelled/deleted occurrences are notoriously hard to detect in Graph (reliable cancellation data only in beta fields). Cross-provider round-tripping (Outlook↔Google) has well-documented failures: COUNT-vs-UNTIL quirks, moved instances beyond series end, and lost exceptions.
4. **Timezones are the classic "event shifted by an hour" bug, and the UK is squarely exposed.** Microsoft Graph uses Windows time zone IDs ("GMT Standard Time") with no clean 1:1 IANA mapping; Google/ICS use IANA/TZID. UK events created near the March spring-forward / October fall-back boundaries, and recurring events spanning a transition, are the prime failure cases.
5. **Rate limits are real but manageable read-only; they bite on two-way at scale.** Google: default 1,000,000 queries/day/project (limits were updated as of 1 May 2026 — verify at build time), per-user limits, 403/429 with exponential backoff required. Microsoft Graph: token-bucket throttling, 429 with authoritative `Retry-After`, ~4 concurrent requests per mailbox and 10,000 requests / 10 min per user/app.
6. **Two-way sync is a step-change in complexity** — echo/loop prevention, conflict resolution, ID mapping, delete propagation, idempotency — which is why scheduling companies (Cronofy, Nylas) sell "don't build this yourself."

## Details

### 1. Webhook reliability

**Google Calendar push (watch channels).** You call `events.watch` with a channel `id`, an HTTPS `address` (callback), and an optional `token`. The callback domain must be verified in Google Cloud console and serve a valid SSL cert. Critically, per Google's official docs, *"Notification messages posted by the Google Calendar API to your receiving URL do not include a message body. These messages do not contain specific information about updated resources, you will need to make another API call to see the full change details."* So every push must trigger an incremental `events.list` with your stored `syncToken`.

Channels expire and *"Currently, there's no automatic way to renew a notification channel. When a channel is close to its expiration, you must replace it with a new one by calling the watch method"* with a new unique `id`. The empirically observed maximum lifetime for Events channels is about 7 days — Google's own `google-api-nodejs-client` GitHub issue #774 states *"Each channel for push notifications expires every 7 days."* Google's official docs decline to publish an exact number, stating only that the expiration is *"a value determined either by your request or by any Google Calendar API internal limits or defaults (the more restrictive value is used)"* — i.e., if you request 30 days, Google silently caps it. Miss the renewal and the app *"goes quiet with no error."* Notifications are also at-least-once (duplicates expected → dedupe) and can be silently dropped; even Google's guidance and community write-ups stress a polling fallback because *"push notifications are not guaranteed for every change."* The `X-Goog-Channel-Token` you set is echoed back and must be validated; a mismatch is a common bug class. Note also the "overlap" window when rotating channels (old + new both active briefly).

**Microsoft Graph change notifications (subscriptions).** A subscription names a resource (e.g., `/me/events`), change types, an HTTPS `notificationUrl`, and a secret `clientState`. On creation Graph performs a validation-token handshake (you must echo the token). Subscriptions have a hard maximum lifetime that for calendar events is ~3 days: the documented/observed cap is **4,230 minutes** (Microsoft's Graph error message, per Microsoft Q&A: *"Subscription expiration can only be 4230 minutes in the future"*), and per Microsoft Learn's subscription resource reference *"any request with expirationDateTime set to under 45 minutes after the time of the request is automatically set to 45 minutes."* You must renew before expiry (PATCH with new `expirationDateTime`) or recreate. Basic notifications carry only the resource `id` — you re-fetch. Graph explicitly supports **lifecycle notifications** (`reauthorizationRequired`, `subscriptionRemoved`, and `missed`) precisely because notifications can be lost: a `missed` event means *"Graph could not deliver some notifications, so you run a delta query or list to backfill the gap."* Delivery is at-least-once (Event Hubs/webhook), so idempotency is required. A significant constraint: subscriptions to `/me/events` require **delegated** permissions (a signed-in user / valid token); app-only subscriptions to a user's mailbox/calendar are not supported — so keeping subscriptions alive when users are signed out is a real operational problem.

**Delta/sync-token as reconciliation.** Microsoft's own guidance and MVP write-ups converge on a hybrid: webhooks for the fast path, delta query for reconciliation, plus *"a scheduled process that runs on a long interval such as 6/12/24/48 hours to trigger your process... to catch any gaps in an active subscription or where a webhook notification wasn't fired or received."* Also renew subscriptions proactively (check expiry, renew when close).

**General pattern & recommended intervals.** Every serious source treats webhooks as latency optimizations layered over a reconciliation loop. Practical fallback polling cadences reported: ~15-minute polling fallback for near-real-time products, plus a longer (6–24h) full/delta sweep to catch anything missed. For a venue system, a 15-min delta reconciliation plus a nightly full sweep is a sound baseline.

### 2. Recurring-event edge cases

**Representation differs by provider.** Google stores a **series master** with a `recurrence` array (RRULE/EXDATE/RDATE, RFC 5545). To get concrete occurrences you pass `singleEvents=true` (which expands the series) — and crucially the `singleEvents` value must be *identical* across the initial and incremental syncs or the API returns 400. Microsoft Graph returns a `seriesMaster` from `/me/events` and does **not** generate occurrences there; you must call `/me/calendar/calendarView?startDateTime=…&endDateTime=…` or `/events/{id}/instances` to expand within a window (default page size 10). Graph's `type` field is one of `singleInstance`, `occurrence`, `exception`, `seriesMaster`.

**Exceptions and cancellations.** When a single occurrence is edited (time/subject/etc.), Graph promotes it to `type: exception` linked via `seriesMasterId`. **Cancelled/deleted occurrences are a known pain point:** the `cancelledOccurrences` and `exceptionOccurrences` properties are documented on v1.0 but *"only work in the beta API"*; deleted instances aren't returned at all, and developers report `isCancelled:true` occurrences not appearing in `calendarView`. Recommendation from Microsoft's own Q&A: cancel (not delete) occurrences, use beta `$expand=exceptionOccurrences`, and maintain your own list of exception/cancelled dates per series. Another Graph gotcha: `calendarView`/`events` only return a recurring series if the query window includes the **series start date** in some configurations, per developer reports.

**Interoperability failures (Outlook↔Google).** The OutlookGoogleCalendarSync project's issue tracker is a catalogue of real-world breakage: recurrence ended by *number of occurrences* (COUNT) failing to sync when end-by-date (UNTIL) works; moved instances failing to appear ("Failed to find master Google event"); sync failing after adding a recurrence exception ("Failed to retrieve master for Google recurring event outside of sync range"). CalendarBridge documents that *"Google will let you move an occurrence of a recurring event beyond the original end date of the series. Microsoft does not support this."* iCalendar itself historically lacked clean exception handling (EXDATE/EXRULE are the mechanism, but tooling support varies).

**Implication for venue availability.** A recurring external booking (e.g., a weekly rehearsal) that your system fails to expand correctly — or a single cancelled occurrence you don't detect — silently corrupts availability: you either block a slot that's actually free (lost revenue) or free a slot that's actually taken (double-booking). For a venue, **always expand recurrences into concrete, dated instances in your own store**, key each instance to its master + original start, and reconcile cancellations explicitly rather than trusting a delta to surface them.

### 3. Timezone & DST — with UK BST/GMT emphasis

**Representation.** Google events use `start.dateTime` + `start.timeZone` (IANA, e.g., `Europe/London`) or `start.date` for all-day. ICS uses `DTSTART;TZID=Europe/London:…` referencing a `VTIMEZONE` block. Microsoft Graph uses a `dateTimeTimeZone` object where the zone is typically a **Windows** ID ("GMT Standard Time", not "Europe/London"), though Graph also accepts IANA on input and you can pin output with `Prefer: outlook.timezone`. `mailboxSettings.timeZone` returns Windows-format strings too.

**The Windows↔IANA mapping problem.** There is *no one-to-one mapping* — one Windows zone maps to multiple IANA zones, and the mapping changes as DST rules change. Microsoft's own SDK issue trackers carry long-standing requests to expose conversion helpers; teams end up maintaining CLDR mapping tables. Browsers/most systems speak IANA; Graph speaks Windows — the seam is a recurring bug source.

**DST edge cases.** The classic bug: an event created before a DST change displays an hour off after it. Documented UK-relevant failure: creating events with `"GMT Standard Time"` where the API doesn't apply BST correctly, so the event edits show as an hour behind ("the time is not taking daylight savings into account"). A subtle Graph trap: **appending `Z` to a timestamp forces UTC and ignores your `timeZone` field.** For ICS→Google, if a `TZID` lacks a matching valid `VTIMEZONE` block, *"Google resolves this using its own rules. If those rules differ from yours, the stored event diverges immediately"* — recurring events are *"the most common source of timezone complaints"* because RRULE expansion interacts with DST. Floating times (no zone) inherit the viewer's/runner's zone — dangerous for automation.

**Recommended practice for a UK venue.** Store the canonical instant in **UTC**, plus the **IANA zone** (`Europe/London`) and the original local wall-clock time. Expand recurrences using the IANA zone so BST/GMT transitions are applied per-occurrence (an event at "19:00 every Friday" stays 19:00 local across the transition, but its UTC offset shifts). Never store `Europe/London` as a fixed offset. For all-day/date-only events, treat them as floating dates (no zone conversion) to avoid off-by-one-day errors. Test explicitly with events straddling the last Sunday of March and last Sunday of October.

### 4. Rate limits & quotas

**Google Calendar API.** Per Google's Usage limits page, the default project ceiling is **1,000,000 queries/day**, with per-minute and per-user limits; *"If either quota is exceeded, you will be rate limited and receive a 403 or 429 status code."* Google's quota page also notes *"As of May 1, 2026, the usage limits for this API were updated"* — verify current numbers at build time. 403/429 carry reason `rateLimitExceeded`/`userRateLimitExceeded` — Google states these *"are functionally similar and should be responded to in the same way, by using exponential backoff"* (with jitter). A common real-world trap: staying under daily quota but still hitting *per-user* rate limits when one account drives many requests (expanding recurrences per calendar is a frequent culprit). Mitigations: exponential backoff, `quotaUser`, service accounts with domain delegation, and push notifications to avoid polling.

**Microsoft Graph.** Throttling is a token-bucket over request costs; over the limit returns **429 Too Many Requests** with an authoritative **`Retry-After`** header (*"Do not guess, do not subtract... Sleep for at least that long"*). Outlook limits apply per **app + mailbox**: per Microsoft's own dev-blog author, *"The limit is 10000 requests per 10-minute period, per user (or group), per app ID"* (≈16–17/s, but Microsoft recommends designing for 4–10/s), and Microsoft Q&A confirms *"Only 4 concurrent requests are allowed at any given time"* per mailbox. Subscription writes are limited to ~500 requests / 20s / app / tenant (notifications don't count). Limits are **not raisable** for calendar; Microsoft points high-volume scenarios to Graph Data Connect. Note some Outlook endpoints return 429 with `Retry-After: 0` or omit the header on certain resources — fall back to exponential backoff.

**ICS feed polling / refresh reality.** Hosted feeds refresh slowly and unpredictably. **Google re-fetches subscribed ICS URLs roughly every 12–24 hours** — Google's own Calendar Help wording has drifted over time from *"every few hours"* to *"up to 8 hours"* to *"up to 12 hours"* (per Cheqroom's Help Center notes), and multiple guides report *"Google Calendar checks subscribed feeds every 12 to 24 hours. There is no user setting to make it faster. There is no publisher setting either."* There is **no way to force a refresh**, and timing varies by subscriber count and change frequency. **Outlook on the web / Microsoft 365 feeds refresh every ~1 to 4 hours** depending on account type and load, and Microsoft's subscribe-or-import guidance notes updates can take *up to 24 hours* worst case. OTA feeds (Airbnb/Booking) are similar: platforms *"refresh imported iCal feeds every 1 to 6 hours, not in real time."* If *you* are the one polling a source ICS feed (rather than relying on Google's subscription), you control cadence — poll politely (send `If-Modified-Since`/honor `Last-Modified`, cache, back off on errors), typically every 15–60 min. Beware CDN/caching staleness on the *source* side regardless of your poll rate.

### 5. Read-only feeds vs. two-way sync

**Read-only ingestion patterns.**
- **ICS feed polling:** simplest possible integration — fetch a `.ics` URL on a timer, parse VEVENTs, map to busy blocks. One-way, read-only, no OAuth, no webhooks. Staleness is the defining limitation (hours). Good for: displaying external busy times, soft availability warnings, low-stakes calendars.
- **One-way API pull with sync tokens (Google) / delta (Microsoft):** requires OAuth and more code, but gives near-real-time, incremental, reliable reads with proper change/delete detection. This is the right read-only path when accuracy matters.

**Two-way sync — the hard parts.**
- **Sync token / delta link management:** persist one token *per calendar* (a user with 8 calendars = 8 tokens + 8 fallback paths). Google returns **410 GONE** (`fullSyncRequired`) when a token is invalid → *"wipe the store and re-sync."* Microsoft returns 4xx (`syncStateNotFound`) → re-seed. Tokens only appear on the *final* paginated page.
- **Echo/loop prevention:** the cardinal two-way bug is your own write bouncing back as an inbound change, looping forever. Solutions: tag events you create with metadata (Google **extended properties** — private key/value pairs hidden on the event; Microsoft **`singleValueLegacyExtendedProperty`**, though developers report reliability caveats and some fall back to encoding a marker in the subject/location), or track a mapping table of external↔internal IDs and skip changes you originated. Use `transactionId` (Graph) for create idempotency.
- **Conflict resolution:** last-write-wins is simple but loses data; field-level merge or version/`changeKey`/`lastModifiedDateTime` comparison is more correct. Compare business fields (start/end/subject/location/cancellation), not just system version fields, to collapse noisy technical updates into one logical change.
- **Deletes/cancellations:** the trickiest to propagate correctly, especially per-occurrence cancellations (see §2). Decide soft-delete vs hard-delete and asymmetric rules explicitly.
- **ID mapping:** external IDs are not always stable (Graph itemIds change on folder move → use **immutable IDs**; Google event IDs are stable). Store the provider, calendar ID, event ID, series master ID, and your internal booking ID.

**The complexity step-change (evidence).** Building calendar sync is widely described as one of the hardest common integrations. Nylas states plainly that *"Building an integration for one service provider from scratch can take as long as 12 months for a team of four engineers."* Cronofy frames it as *"you wouldn't try to rebuild Stripe for your payments, so why try and rebuild scheduling infrastructure."* Truto states *"calendar integrations are notoriously hostile to build, scale, and maintain"* and estimates *"Building a basic Google Calendar or Outlook integration from scratch typically costs between €1,500 and €2,500 per provider just for the initial MVP phase"* (rising past €80,000 per provider for enterprise-grade). The recurring theme: read-only pull is a bounded, well-understood task; two-way sync introduces loop prevention, conflict resolution, and delete propagation that each spawn edge cases.

**Unified API providers (Nylas, Cronofy, etc.) — worth it?** They abstract multi-provider OAuth, normalize event models, handle timezone/DST and recurrence expansion, and manage webhooks/renewals. Trade-offs: **cost** (Cronofy from ~$15/user/mo; Nylas from ~$10/mo for calendar-only with 5 accounts — material at 1,000+ connected calendars, model against ARPU), **control/data-residency** (Merge caches data; Cronofy/Truto emphasize minimal retention — relevant for UK GDPR), and **vendor lock-in / migration risk** (Nylas forced a painful v3 migration; Cronofy touts a decade of v1 stability). Cronofy is scheduling/availability-focused (strong for a venue), Nylas bundles email+calendar+contacts. For a UK venue with a small team wanting two-way sync fast and GDPR-friendly hosting, a unified provider is a defensible buy; for Google-only or read-only scopes, direct APIs are cheaper.

### 6. Synthesis & recommendations for a venue system

**Recommended P3 architecture (direct integrations):**
1. **Ingestion tier per provider.** Google: OAuth + `events.watch` push → on ping, run incremental `events.list` with stored `syncToken` (`singleEvents=true`, consistent across syncs). Microsoft: OAuth (delegated) + Graph subscription on `/me/events` with `clientState` + `lifecycleNotificationUrl` → on notification, run delta query; expand via `calendarView` for availability windows.
2. **Webhooks as hints + reconciliation as truth.** Renewal jobs: recreate Google channels before ~7 days; renew Graph subscriptions before ~3 days (4,230 min). Reconciliation: 15-min delta/sync-token sweep + nightly full sweep. Handle 410/`syncStateNotFound` → full resync. Dedupe all notifications (at-least-once).
3. **Data model.** Per external event store: `provider`, `calendar_id`, `external_event_id` (immutable ID for Graph), `series_master_id`, `original_start`, `recurrence_rule`, `is_cancelled`, `etag/changeKey`, `last_modified`, `sync_token`/`delta_link`, and `internal_booking_id`. Store times as **UTC instant + IANA zone + local wall time**. Tag self-created events with extended properties for echo prevention.
4. **Recurrence handling for availability.** Expand every series into concrete dated instances in your store; never compute availability from an unexpanded master. Reconcile cancelled occurrences explicitly (poll `calendarView`/instances; use Graph beta cancellation fields if needed). Re-expand across DST boundaries using the IANA zone.
5. **Availability integrity.** Availability queries read your expanded-instance store, not the live API. A confirmed internal booking is authoritative; external feeds contribute *busy* blocks. Add booking guardrails (buffer time, minimum-advance-notice) to cover the sync-delay window, mirroring OTA best practice.

**Verdict on ICS read-only as the "cheap early win": YES, with guardrails.**
- **What it delivers:** a genuinely fast, low-cost way to pull external busy times (staff Google calendars via "secret address" ICS, Outlook published calendars, OTA feeds) into the venue view and to soft-warn on conflicts. No OAuth, no webhooks, no renewal jobs. This is real value and ships in days, not weeks.
- **Effort vs. full API:** roughly an order of magnitude less than a single two-way API integration (an ICS poller + parser vs. OAuth + webhook + sync-token + renewal + reconciliation). It is a legitimate Phase-0 that de-risks the data model (you learn your event/timezone/recurrence mapping cheaply). For calibration, Truto pegs even a *basic* one-way API MVP at €1,500–€2,500 per provider — an ICS poller is well below that.
- **Fidelity/staleness limits that must be designed around:**
  - **Staleness:** if you rely on Google *subscribing* to a feed, refresh is ~12–24h and unforceable; Outlook/M365 feeds ~1–4h (up to 24h). If *you* poll the source feed directly, you control cadence (15–60 min) — strongly prefer this.
  - **No change notifications:** ICS gives you no signal that something changed; you only know by re-fetching and diffing.
  - **Recurrence/timezone fidelity:** ICS recurrence and TZID handling is inconsistent; missing/invalid `VTIMEZONE` causes hour drift; exceptions may be incomplete. Parse defensively and expand yourself.
  - **One-way only:** cannot write back — fine for "show external busy," useless for confirming bookings externally.
- **Bottom line:** ship ICS read-only to display external busy time and *warn* on conflicts, but **do not let a stale ICS feed be the gate that confirms or hard-blocks a paid booking.** Confirmed bookings must live in your own authoritative store, backed by the API integrations in P3.

## Recommendations

**Stage 0 (now, days): ICS read-only pilot.** Build a polite ICS poller (honor `Last-Modified`, cache, 15–60 min cadence, back off on errors). Parse VEVENTs into busy blocks; expand recurrences yourself; store UTC + IANA zone. Surface external busy time in the venue view and soft-warn on overlaps. *Do not* hard-block bookings on it. **Threshold to advance:** once double-booking risk from staleness becomes unacceptable, or you need write-back, move to Stage 1.

**Stage 1 (P3, weeks): direct Google + Microsoft read integrations.** OAuth + sync token (Google) / delta query (Microsoft), webhooks-as-hints with 15-min delta reconciliation + nightly full sweep, channel/subscription renewal jobs, 410/expiry handling, dedupe. This gives near-real-time, reliable external availability. **Threshold to advance to two-way:** only when the product genuinely needs to *write* venue bookings back into users' calendars.

**Stage 2 (later, only if needed): two-way sync.** Add echo prevention (extended properties + ID mapping), conflict resolution (field-level, `changeKey`/`etag`), idempotency (`transactionId`), and explicit delete/cancellation propagation. **Decision gate:** if two-way is needed and team bandwidth is under roughly one engineer-quarter per provider, **buy** a unified provider (Cronofy preferred for scheduling focus + GDPR/EU hosting; Nylas if you also need email/contacts) rather than build. Re-evaluate build-vs-buy against per-connected-calendar cost vs. ARPU.

**Cross-cutting must-dos:**
- Store UTC + IANA `Europe/London` + local wall time; never fixed offsets. Add automated tests for events straddling the March and October UK transitions.
- Expand all recurrences into dated instances; reconcile cancellations explicitly.
- Treat every webhook as an at-least-once hint; reconcile on a timer.
- Add booking guardrails (buffers, min-advance-notice) to absorb sync latency.

**Highest-risk pitfalls, ranked by likelihood × impact (UK venue):**
1. **DST ±1h drift (BST↔GMT)** — high likelihood, high impact (mis-timed bookings). Mitigate with UTC+IANA storage and boundary tests.
2. **Recurring-event / cancelled-occurrence mishandling** — high likelihood, high impact (silent availability corruption). Mitigate by expanding to instances and explicit cancellation reconciliation.
3. **Double-booking from stale ICS or dropped webhooks** — medium-high likelihood, high impact. Mitigate with reconciliation loops, booking guardrails, and never gating on ICS.
4. **Sync-token/subscription expiry going silent** — medium likelihood, high impact (sync quietly dies). Mitigate with proactive renewal + 410/`syncStateNotFound` handling + monitoring/alerting.
5. **Rate-limit throttling at scale** — medium likelihood, medium impact. Mitigate with backoff, `Retry-After` compliance, batching, and per-user load distribution.
6. **Echo loops / duplicate bookings in two-way** — medium likelihood (only if two-way), high impact. Mitigate with origin tagging + idempotency.

## Caveats
- **Numbers drift.** Google's ~7-day channel lifetime is community-observed (GitHub issue #774, Nylas, tutorials), *not* an officially published figure — Google documents only that it caps to an internal limit. Microsoft's 4,230-minute calendar-subscription cap is well-attested but Microsoft can change it. Google's 410 sync-token window has changed silently before, and Google's per-project quota was updated as of 1 May 2026. Verify current values against live docs at build time.
- **Docs vs. reality diverge** most on: Graph cancelled-occurrence data (v1.0 docs list fields that only work in beta), ICS refresh timing (undocumented and drifting — Google's own help text has changed its stated figure repeatedly), and per-mailbox Graph limits (some endpoints omit `Retry-After`).
- **Vendor blog sources have bias.** Cronofy/Nylas/SYNCDATE/CalendarBridge/Truto content is partly marketing for "buy, don't build." Their *technical* descriptions of provider behavior corroborate primary docs and are cited accordingly, but treat cost/effort framing as directional.
- **Scope note.** This analysis assumes Google Calendar and Microsoft 365/Outlook as the primary external providers plus generic ICS; Apple/iCloud (CalDAV) and Exchange on-prem (EWS) have additional quirks not covered here.
